# UI 피드백 검토 및 웨이브 개선 계획

> 점검일 2026-07-19 · 대상 prompt-gallery.byungwook-an.workers.dev · 배포 사이트 + 소스 교차검증

사용자 육안 피드백과 앞선 AI 진단(Artifact)을 대조하고, 코드로 원인을 검증한 뒤 웨이브로 정리했습니다.
**이 문서는 검토·계획이며, 코드는 아직 수정하지 않았습니다.**

## 1. 중복 체크 — 사용자 피드백 vs AI 진단

| # | 사용자 피드백 | AI 진단 항목 | 관계 |
|---|---|---|---|
| A | 상세모달 "복사됨" 줄 밀림 | (없음) | 🆕 사용자만 지적 |
| B | 즐겨찾기 반응 느림 | (없음) | 🆕 사용자만 지적 (성능) |
| C | 모달 뱃지 찌그러짐 | 🔴 1번 뱃지 세로 파괴 | ✅ **중복 — 동일 이슈** |
| D | 제목 2군데 중복 | 🔴 2번 본문 2번 반복 | ✅ **중복 — 동일 이슈** |
| E | 제목 없을 때 1줄로 | 🔴 2번의 일부 | ⚠️ **부분 중복 + 새 요구** |
| F | 화면 접으면 내보내기 버튼 내려감 | 🟠 5번 탭바 깨짐 | ⚠️ 유사(반응형)하나 **다른 요소** |
| G | 즐겨찾기 탭 → 오른쪽 / All 왼쪽 | (없음) | 🆕 정보구조 재배치 |
| H | All에서 masonry 통합 뷰 | 🟠 6번 빈 섹션 점유 | ⚠️ 6번을 **포함·확장**하는 상위 해법 |
| I | 레포 제목 `owner/repo` | 🟠 4번 레포 제목 잘린 URL | ✅ **중복 — 동일 이슈** |
| J | 레포 본문에 README 표시 | (없음) | 🆕 추가기능 |
| K | 출처/링크 필드 추가 | (없음) | 🆕 추가기능 |
| L | 파비콘 "P" 심볼로 | (없음) | 🆕 브랜딩 |

**중복 정리**
- **완전 중복 (C·D·I)**: 사용자와 AI가 같은 이슈를 지목. 한 번만 처리.
- **부분/포함 (E·H)**: E는 D와 얽혀 있고, H는 6번(빈 섹션)을 근본적으로 해소하는 상위 해법. → **H를 하면 6번은 자동 해결**.
- **AI 진단 중 사용자가 언급 안 한 것**: 🔴 3번(정사각 카드 여백)은 사용자 피드백엔 없지만 여전히 유효. H(masonry)로 함께 해소 가능.
- **사용자만 지적 (A·B·G·J·K·L)**: AI가 놓친 부분. 특히 **B(성능)**는 정적 스크린샷 진단의 사각지대였음.

## 2. 코드 검증 결과 (원인 확정)

각 이슈를 소스에서 확인해 원인과 작업 성격을 특정했습니다.

### 확정된 원인

| 이슈 | 원인 (파일:라인) | 성격 |
|---|---|---|
| C 뱃지 찌그러짐 | `.type-badge`에 `flex:0 0 auto`·`white-space:nowrap` 없음 (cards.css:81) | CSS 1~2줄 |
| A 복사됨 줄밀림 | 모달 `<output class="modal-toast">` 별도 블록 렌더 (ItemModal.tsx:234) | JSX 삭제 |
| D 제목 중복 | 카드가 `preview`(본문)+`h3`(title) 둘 다 렌더. title이 본문 복제본 (GalleryCard.tsx:162-163) | 로직 |
| B 즐겨찾기 지연 | 토글 후 **items·tags·workflows 3개 API 전체 재fetch** (App.tsx:88·138) + 모달은 `disabled={saving}`로 응답 대기 | 성능/구조 |
| I 레포 제목 | 서버가 title 미지정 시 URL 전체를 title로 저장 (item-input.ts:127-129) | 로직(FE/BE) |
| E 제목 1줄 | 카드·모달 `h3`가 2줄 말줄임 (cards.css:182 `-webkit-line-clamp:2`) | CSS (마이그레이션 불필요·확정) |

### 유리하게 작용하는 기존 자산

- **`contentUrl` 필드가 이미 존재** — `gallery-data.ts:25`, worker의 `item-types`·`item-routes`·`export`에 배선 완료. → **K(출처 링크) 추가기능의 백엔드 절반은 이미 구현됨**. UI 폼/표시만 붙이면 됨.
- 반면 **J(README 표시)**는 GitHub API 호출 + 캐싱이 필요 → worker 신규 작업. K와 난이도 차이가 큼.

### E(제목 1줄) — 결정 완료

당초 "제목 없을 때만 1줄"로 이해해 데이터 모델(파생/지정 구분) 문제로 봤으나,
**사용자 확정: 지정이든 자동복사든 무조건 1줄 표시.** → 구분 불필요, 마이그레이션 불필요.
카드·모달 `h3`를 `-webkit-line-clamp:1`로 바꾸는 CSS 변경으로 끝. (Wave 3에서 D와 함께)

단, 제목 1줄만으로는 **D(제목=본문 첫 줄 중복)**가 남음: 제목이 본문 복사본이면
본문 미리보기 첫 줄과 제목이 같은 문장으로 겹쳐 보임. → D를 Wave 3에서 함께 처리.

## 3. 웨이브 제안

성격·리스크·의존성 기준으로 4개 웨이브. **Wave 1·2는 저리스크 즉효, Wave 3은 구조 변경, Wave 4는 신규 기능**으로 분리했습니다.

### 구현계획 문서 분할 (2026-07-19 확정)

웨이브 번호가 아니라 **리스크·의존성·변경 표면** 기준으로 2개 구현계획 문서로 나눈다.

| 구현계획 문서 | 범위 | 근거 |
|---|---|---|
| **구현계획 A** | Wave 1 + 2 + 3 | 전부 클라이언트 전용·마이그레이션 없음·한 배포 단위. 2의 상태헬퍼 → 3의 레이아웃이 이어지는 의존 흐름 |
| **구현계획 B** | Wave 4 (+ Wave 5를 후속 절로) | worker·D1·외부(GitHub) 연동. 리스크·검증 방식이 A와 완전히 다름. Wave 5(규칙 반영)는 1~4 전체 종료에 종속되므로 B 말미 절로 배치 |

**진행 순서**: A 인터뷰 → 구현계획 A 작성 → A 구현·배포·검증 → B 인터뷰 → 구현계획 B 작성 → B 구현 → Wave 5 diff 승인.
이 문서(검토·진단)는 정본 유지, 구현 상세(파일·함수·완료기준·검증)는 A/B 문서에 별도 작성.

### Wave 1 — 즉시 수정 (CSS·JSX 소규모, 데이터 무관)
> 리스크 낮음 · 빠른 체감 개선 · 마이그레이션 없음

- **C** 모달 뱃지 1줄 고정 (`flex:0 0 auto; white-space:nowrap`)
- **A** 모달 "복사됨" 토스트 줄 삭제 (아이콘이 이미 체크로 바뀜)
- **3(AI)** 정사각 카드 여백 → `aspect-ratio` 제거, `min-height`만 (H로 흡수 가능하나 독립 처리도 가능)
- **L** 파비콘 "P" 심볼로 교체

### Wave 2 — 성능 (즐겨찾기 즉시 반응)
> 리스크 중간 · B 단독. 사용자 체감 가장 큰 항목

- **B** 즐겨찾기 optimistic update
  - 토글 즉시 로컬 상태 반영 → 서버는 백그라운드 전송
  - 실패 시에만 롤백
  - 모달 버튼 `disabled={saving}` 제거 (즉각 반응)
  - 전체 3-API 재fetch 대신 해당 아이템만 갱신

### Wave 3 — 정보구조·레이아웃 재편 (구조 변경)
> 리스크 중상 · 화면 구성이 바뀜 · 상호 의존

- **G** 탭 순서: `All`(기본) 최좌측, `즐겨찾기` 최우측으로 재배치
- **H** All 탭 = masonry 통합 뷰 (카테고리 가로 분할 제거, 카드 뱃지로 구분)
  - → AI 6번(빈 섹션)·3번(정사각) 자동 해소
- **F** 반응형: 접었을 때 "Prompt Gallery" 타이틀 숨김, 버튼 1줄 유지
- **D+E** 제목 중복 제거 + 제목 없을 때 1줄 (Wave 3에서 함께, 단 E는 §4 결정 선행)
- **I** 레포 카드 제목 `owner/repo`로 표시

### Wave 4 — 추가 기능 (신규 필드·외부 연동)
> 리스크 높음 · 스펙 정의 필요 · 독립 배포 가능

- **K** 출처/링크 필드 (`contentUrl` 활용 — 백엔드 기반 존재)
  - 모든 타입에 "출처 링크" 입력·표시
  - 레포도 소개/출처 링크 별도 지정
- **J** 레포 본문에 README 표시 (GitHub API + 캐싱, worker 신규)
  - ⚠️ rate limit·비공개 레포·캐싱 정책 결정 필요
- **I-확장** 레포 제목 "의미 단위" 정리 (README 제목/repo description 활용 — J와 연동)

### Wave 5 — 프로젝트 규칙 반영 (모든 구현 종료 후)
> 재발 방지 성문화 · **diff 제안 → 사용자 승인 후 반영** · 규칙 파일 직접 편집 안 함

- **반영 대상 파일**: `src/client/AGENTS.md` (이미 `## CONVENTIONS`·`## ANTI-PATTERNS` 섹션 존재 → 신규 문서 불필요)
  - 결정(2026-07-19): **CLAUDE.md 생성 안 함**. Claude Code가 AGENTS.md를 직접 인식하므로 실익 없고 동기화 부담만 발생. AGENTS.md 단일 정본 유지 (session-dashboard 선례와 동일).
- **추가할 규칙 (최소 문장)**:
  - ANTI-PATTERNS: "쓰기 동작 후 전체 갤러리 재fetch 금지. 로컬 상태 부분 갱신(optimistic) + 실패 시 롤백을 기본으로." (Wave 2 근거)
  - CONVENTIONS: "네트워크 에러 처리·요청 전송은 공용 api-client를 재사용. mutation 파일마다 복붙 금지." (Wave 2-e 근거)
  - CONVENTIONS(반응형 회귀 방지, 2026-07-19 사용자 요청): "UI 변경 시 상단 액션 버튼이 어느 뷰포트에서도 단일 행을 유지하는지, 라벨·버튼이 의도치 않게 2줄로 wrap되지 않는지 확인. qa:browser의 `assertSingleRow`가 topbar-actions를 자동 검사하며, 새 버튼 그룹 추가 시 동일 검사를 적용." (Wave 3.5 근거)
- **⚠️ 함께 수정 필요한 기존 규칙 (코드-규칙 정합)**:
  - `AGENTS.md` DESIGN RULES 36-37줄의 "square card grid" 규칙 → Wave 3에서 masonry 통합뷰(H)로 바꾸면 이 규칙이 코드와 어긋남. Wave 5에서 함께 갱신.
- **절차**: Wave 1~4 완료 후, 위 변경을 **diff 형식으로 제안** → 사용자 승인 → 반영. AGENTS.md는 병행 작업 파일일 수 있어 임의 편집하지 않음.

## 4. 추가로 짚는 이슈 (사용자·AI 모두 미언급)

검토 중 발견한, 정리가 필요한 지점들:

1. **즐겨찾기 탭 자체가 동작하는가** — 탭 재배치(G) 전에, 현재 즐겨찾기 필터가 정상 동작하는지 확인 필요 (즐겨찾기된 아이템이 0개라 미검증).
2. **카드 태그 "+Tag" 버튼이 비활성** — 태그 0개 상태에서 disabled. Wave 3~4에서 태그 UX와 함께 볼지 결정.
3. **접근성** — masonry(H) 전환 시 `column-count` 기반이면 탭 순서가 시각 순서와 어긋날 수 있음. 키보드 내비 검증 항목 추가.
4. **README 렌더링 안전성 (J)** — 마크다운을 그대로 표시할지, 텍스트만 뽑을지. HTML 렌더 시 XSS 고려.
5. **contentUrl 마이그레이션 상태** — 필드는 있으나 실제 원격 D1에 컬럼이 적용됐는지 배포 확인 필요 (최근 커밋이 migration 관련이라 점검 권장).

## 5. 결정이 필요한 항목 (인터뷰 대상)

### 확정된 사항 (2026-07-19 인터뷰)

- **E**: 제목은 지정/자동 무관 **무조건 1줄**. 마이그레이션 없음.
- **진행 방식**: Wave 1 → 2 → 3 → 4 **순차 진행**.
- **인터뷰**: 각 웨이브 **시작 전 상세 스펙 인터뷰** 후 착수.

### Wave별 착수 전 결정할 항목

- **Wave 3**: D(제목=본문 중복) 처리 방식 / masonry 레이아웃(column 기반 vs grid) / F 반응형 브레이크포인트
- **Wave 4**: J(README) 진행 여부·렌더 방식(마크다운 vs 텍스트) / K 출처 필드를 모든 타입에 노출할지 / contentUrl 원격 D1 마이그레이션 적용 확인
