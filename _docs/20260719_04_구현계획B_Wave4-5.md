## 1. 재사용 가능한 기존 자산 (신규 구현 최소화)

착수 전 확인한, K 구현이 그대로 본뜰 수 있는 기존 배선:

- **`github_url` 컬럼의 전체 배선 경로** — K(`source_url`)는 이 경로를 그대로 복제하면 된다:
  1. `migrations/0001_initial.sql` items 테이블 (컬럼 정의)
  2. `item-types.ts` — `Item.githubUrl`, `ItemResponse`, `ItemRow.github_url`, `CreateItemInput`, `PatchItemInput`, `rowToItem`, `itemResponse`
  3. `item-input.ts` — `readOptionalString(value, "githubUrl")` (create·patch)
  4. `item-repository.ts` — `SELECT_ITEM`(18행), INSERT(94행), UPDATE `addStringField`(140행)
  5. `gallery-data.ts` — 클라이언트 파싱(`readOptionalNullableString`)
  6. `ItemModalForm.tsx` — 입력 폼
  7. `GalleryCard.tsx` / `ItemModalDetail.tsx` — 표시
- **`itemResponse`가 export에도 사용됨**(`export-routes.ts:48`) → `ItemResponse`에 `sourceUrl` 추가하면 **export JSON에 자동 포함**. export-routes 별도 수정 불필요.
- **`addStringField` 헬퍼**(item-repository.ts:213) — undefined면 스킵, 값 있으면 `column = ?` 추가. `source_url`도 이 헬퍼로 처리.

**차이점(github_url과 다르게 처리할 점)**:
- `github_url`은 레포 타입에 **필수·형식검증**(`isValidGitHubUrl`)이 있으나, `source_url`은 **모든 타입 선택 입력**. 형식검증은 "URL 파싱 가능" 정도로 느슨하게(또는 무검증) — 부록 A-1 참조.
- title 자동생성 로직(`deriveCreateTitle`)에 `source_url`은 관여시키지 않는다.

## Wave 4 — K 출처 링크 필드

리스크 중간 · **D1 마이그레이션 동반** · worker+client 양쪽 · 독립 배포 단위.

### 4-0. 마이그레이션 (migration 0003)
- **파일 신규**: `migrations/0003_item_source_url.sql`
- **내용**: `ALTER TABLE items ADD COLUMN source_url TEXT;` (nullable, 기본 NULL)
- **완료기준**: 로컬 D1에 컬럼 추가됨. 기존 행은 `source_url = NULL`
- **검증**:
  - 로컬 적용 `pnpm exec wrangler d1 migrations apply prompt-gallery-db --local` 성공(종료코드 0)
  - `pnpm exec wrangler d1 execute prompt-gallery-db --local --command "PRAGMA table_info(items)"` 출력에 `source_url` 포함
- **⚠️ 원격 반영**: 배포 시 `--remote`로 원격 D1(database_id `138200be-…`)에도 적용 필요. **본 계획은 로컬까지. 원격 적용은 배포 절차(§배포)에서 명시적으로 수행**.

### 4-1. 타입·응답 배선 (worker)
- **파일**: `src/worker/item-types.ts`
- **변경**:
  1. `Item`에 `readonly sourceUrl: string | null` 추가
  2. `ItemRow`에 `readonly source_url: string | null` 추가
  3. `CreateItemInput`·`PatchItemInput`에 `sourceUrl` 추가 (create는 `string | null`, patch는 `sourceUrl?: string | null`)
  4. `rowToItem`에 `sourceUrl: row.source_url` 매핑
  5. `itemResponse`에 `sourceUrl: item.sourceUrl` 추가 (`ItemResponse`는 `Omit<Item,"imageKey">` 기반이라 자동 포함되나 명시 확인)
- **완료기준**: worker 타입이 `source_url` 컬럼을 왕복(round-trip)한다
- **검증**: `pnpm typecheck` 통과

### 4-2. 입력 검증 (worker)
- **파일**: `src/worker/item-input.ts`
- **변경**:
  1. `parseCreateItem`에 `sourceUrl: readOptionalString(value, "sourceUrl") ?? null` 추가
  2. `parsePatchItem`에 `const sourceUrl = readOptionalString(value, "sourceUrl")` + patch 병합(`...(sourceUrl !== undefined ? { sourceUrl } : {})`)
  3. **형식검증**: 값이 있을 때 `new URL(value)` 파싱 시도, 실패 시 `ApiError("invalid_item", "sourceUrl must be a valid URL.", 400)`. http/https만 허용(부록 A-1). 빈 문자열은 기존 `readOptionalString`이 null 처리.
- **완료기준**: 잘못된 URL은 400, 유효 URL·null은 통과
- **검증**: `pnpm test:worker` — item create/patch 테스트에 `sourceUrl` 케이스 추가(유효 URL 저장·조회, 잘못된 값 400, null 허용)

### 4-3. 저장 (repository)
- **파일**: `src/worker/item-repository.ts`
- **변경**:
  1. `SELECT_ITEM`(17행)에 `source_url` 컬럼 추가
  2. `create`의 INSERT(94행) 컬럼·바인딩에 `source_url`/`input.sourceUrl` 추가
  3. `update`에 `addStringField(fields, values, { column: "source_url", value: input.sourceUrl })` 추가(140행 근처)
- **완료기준**: 생성·수정 시 `source_url`이 저장·반환된다
- **검증**: `pnpm test:worker` 통과(4-2 케이스가 repository까지 관통)

### 4-4. 클라이언트 파싱·타입
- **파일**: `src/client/gallery-data.ts`
- **변경**: `Item` 클라이언트 타입에 `sourceUrl: string | null` 추가, 파싱에 `sourceUrl: readNullableString(record, "sourceUrl")` 추가(기존 `githubUrl`과 동일 패턴, 194행 근처. `source_url`은 항상 응답에 실리므로 `readOptionalNullableString`이 아니라 `readNullableString` 사용)
- **완료기준**: `/api/items` 응답의 `sourceUrl`이 클라이언트 모델에 실림
- **검증**: `pnpm typecheck` 통과

### 4-5. 입력 폼
- **파일**: `src/client/ItemModalForm.tsx`, `src/client/item-modal-model.ts`(Draft 타입)
- **변경**:
  1. `Draft`에 `sourceUrl: string` 추가, 초기화/편집 로딩에 반영
  2. 폼에 "출처 링크"(선택) `<input type="url">` 추가 — **모든 타입 공통**(promptLike/repoLike 조건 없이). placeholder 예: `https://... (출처·소개 링크, 선택)`
  3. GitHub URL 필드(레포 전용)와 **별개** — 레포는 두 필드(GitHub URL 필수 + 출처 링크 선택) 공존
- **완료기준**: 모든 타입 추가·편집 모달에 출처 링크 입력칸이 있고, 저장 시 서버로 전송
- **검증** (검수 §3 반영): qa:browser 시나리오(기존 `browser-*.mjs`는 chromium.launch() **headless** 기본)에서 프롬프트·이미지·레포 각각 출처 링크 입력→저장→재조회. 값 유지 여부를 **기존 `assert*` 헬퍼(browser-gallery-search-support.mjs 패턴)로 DOM assertion** → 실패 시 throw로 비0 종료코드. done-check는 **종료코드 0 + 증거 스크린샷**으로 판정(육안 아님). 증거 `.omo/evidence/`

### 4-6. 표시
- **파일**: `src/client/GalleryCard.tsx`, `src/client/ItemModalDetail.tsx`
- **변경**: `sourceUrl`이 있으면 카드·상세에 "출처" 링크 표시(새 탭 `target="_blank" rel="noopener noreferrer"`). 없으면 미표시. 레포의 GitHub 링크와 시각적으로 구분되는 라벨("출처").
- **완료기준**: 출처 있는 항목은 카드·상세에 클릭 가능한 출처 링크, 없으면 아무것도 안 뜸
- **검증** (검수 §3 반영): qa:browser(headless)에서 출처 있는 항목은 링크 요소 존재 + `href`가 저장값·`rel`이 `noopener noreferrer`임을 **DOM assertion**, 출처 없는 항목은 링크 요소 부재를 assertion. 실패 시 throw로 비0 종료코드. 증거 `.omo/evidence/`

### A-1. K 출처 링크 — `source_url` 신규 컬럼 선택

**검토문서(§2)의 최초 전제가 오류였음**: "`contentUrl` 필드가 이미 존재해 K 백엔드 절반이 구현됨". 실제 코드 확인 결과 `contentUrl`은 **이미지 에셋의 콘텐츠 프록시 URL**(`itemResponse`가 `item.imageAssetId`로부터 `/api/assets/{id}/content`를 파생, `item-types.ts:120`)로, 사용자가 입력하는 출처 링크와 **완전히 무관**. K의 백엔드는 전무했음.

| 옵션 | 내용 | 판정 |
|---|---|---|
| **A-1 (선택)** | `source_url` 신규 컬럼 + 전 타입 입력·표시 | ✅ github_url 배선을 그대로 복제. 명확·확장 가능 |
| A-2 | 레포는 기존 `github_url` 재사용, 프롬프트/이미지만 신규 | ❌ 출처와 GitHub URL은 의미가 다름(출처=어디서 가져왔나, github_url=레포 위치). 레포도 "소개 링크" 별도 필요하다고 사용자가 명시. 필드 혼용은 혼란 |
| A-3 | K도 이번에 제외 | ❌ 사용자가 최초 피드백에서 명시적으로 요청한 항목 |

**형식검증 수위**: `github_url`처럼 엄격한 호스트 검증(github.com 강제)은 부적절(출처는 임의 사이트). `new URL()` 파싱 + http/https 프로토콜만 확인하는 **느슨한 검증** 채택. 파싱 실패만 400.

### A-2. J README 표시 — BACKLOG 이관 (이번 제외)

**사용자가 최초 피드백에서 직접 요청한 항목이 맞음**("본문에는 Readme.md의 내용이 표기되면 좋을 것 같다"). 초기 판단에서 "AI 진단에서 넘어온 것"으로 오독했으나 정정함.

그럼에도 **이번 구현에서 제외**하기로 결정. 검토 경로:

| 옵션 | 내용 | 비용 | 판정 |
|---|---|---|---|
| 마크다운 렌더 | README 전체를 마크다운으로 렌더 | 높음: GitHub API rate-limit(비인증 60/h)·D1 캐싱 정책·비공개 레포 분기·**XSS sanitize** | ❌ 운영 정책 다수 |
| 텍스트만 추출 | README 텍스트만 | 중: 코드블록·목록 뭉개짐 → "내용 표시" 취지 훼손 | ❌ 실속 낮음 |

| description+번역 | 한국어 번역 | 중상: 번역 API 키·비용 추가 | ❌ null이면 무용 |
| **BACKLOG 이관 (선택)** | 이번엔 K만, J는 후속 | 최저 | ✅ |

**실측 근거 (2026-07-19 GitHub API 직접 조회)**:
- 사용자 예시 레포 `Canine89/hwpxskill` → `description: null` (아예 없음)

- → description은 ①`null`이 흔하고 ②있어도 영어라 그대로는 실속이 낮고, ③한국어 번역하면 외부 의존(번역 API)이 하나 더 붙어 "가볍다"는 전제가 무너짐.

**결론**: J는 GitHub API 연동(rate-limit·캐싱·인증 토큰 secret)이 본질적으로 무겁고, 저비용 대안(description)은 실측상 실속이 낮음. 별도 스펙 정의가 필요한 작업이므로 BACKLOG로 이관하고, 착수 시 렌더 방식·캐싱·토큰 정책을 별도 결정한다.
