## 관심사 1 — 모달 헤더 재구성 (커밋 1)

### 1-1. 헤더 2줄 + "편집" 라벨 삭제
- **파일**: `src/client/ItemModal.tsx`
- **변경**:
  1. `title` 계산(74행): `mode === "edit" ? "편집" : item.title` → **detail 모드만 제목 h2 렌더**. add/edit 모드는 헤더에 **제목 h2를 아예 렌더하지 않음**(뱃지+액션만). 이유: 편집·추가 폼에 제목 입력칸이 이미 있어 헤더 제목 글씨는 중복(미리보기 초안 확정 = B안). "새 항목"/"편집" 문구도 표시하지 않음.
  2. 헤더 마크업(180-184행): 1줄 `<div>`에 `ModalTypeBadge` + `modal-header-actions`(복사·즐겨찾기·닫기). **detail 모드일 때만** 2줄에 `<h2 id="item-modal-title">{item.title}</h2>` 추가.
  3. **접근성 부작용 처리(필수)**: `<dialog>`(159행)의 `aria-labelledby="item-modal-title"`는 하드코딩이라, edit/add 모드에서 h2를 렌더하지 않으면 참조 대상 없는 고아 속성이 된다. **모드별 동적 분기**: detail 모드는 `aria-labelledby="item-modal-title"` 유지, edit/add 모드는 `aria-labelledby`를 주지 않고 `aria-label`로 접근성 이름 제공(add=`"새 항목"`, edit=`item.title`). 헤더에 글씨는 없어도 스크린리더용 이름은 유지.
- **파일**: `src/client/styles/modal.css`
  4. `.modal-header`를 세로 스택(뱃지줄 / [detail만]제목줄)으로. `.modal-header h2`(49행)에 `-webkit-line-clamp:1` + box orient + overflow hidden 추가(1줄 말줄임, 전체 폭).
- **완료기준**: **상세 모달** 헤더가 [뱃지+아이콘] / [제목 1줄] 2줄 구조. **편집/추가 모달** 헤더는 [뱃지+아이콘]만(제목 글씨·"편집"·"새 항목" 문구 없음). 긴 제목 1줄 말줄임. 편집/추가 모달 `<dialog>`에 `aria-label` 존재(고아 `aria-labelledby` 없음).
- **검증**: qa:browser(headless)에서 (1) 상세 모달 헤더에 제목 h2 존재 + computed `webkitLineClamp==="1"` + 실제 title 텍스트, (2) 편집 모달 헤더에 제목 h2 **부재**(뱃지만), (3) "편집"·"새 항목" 문구 부재, (4) 편집 모달 `dialog`에 `aria-labelledby` 부재 + `aria-label` 존재(고아 참조 없음). 증거 `.omo/evidence/`

## 관심사 4 — 파비콘 폴백 제거 + DESIGN.md 갱신 (커밋 4)

### 4-1. index.html alternate 링크 제거 (결정 B)
- **파일**: `index.html`
- **변경**: `<link rel="alternate icon" href="/favicon.ico">`(11행) **줄 삭제**. `<link rel="icon" type="image/svg+xml" href="/favicon.svg">`(10행)만 유지.
- **근거**: favicon.svg는 이미 신버전(블루+세리프 P). 문제는 구버전 favicon.ico가 폴백으로 잡혀 브라우저가 이를 우선 사용한 것. 폴백 링크를 끊으면 신 SVG가 뜬다. .ico 재생성(도구 설치·변환) 불필요.
- **트레이드오프**: SVG 파비콘 미지원 구형 브라우저(구 IE 등)는 파비콘이 없어짐. 현대 브라우저는 전부 SVG 파비콘 지원하므로 실질 문제 없음. `public/favicon.ico` 파일 자체는 남겨두되 링크만 제거(직접 URL 접근 시엔 여전히 응답).
- **완료기준**: `index.html`에 favicon.ico 참조 없음. favicon.svg만 참조.

## 완료검수(done-check) 기준

이 기준은 미리보기 위젯을 주고받으며 사용자와 확정한 것이다. **done-check는 아래를 증거 기반으로 판정한다. 하나라도 어긋나면 Blocked.** (반복 지적된 항목이라 명시적으로 고정한다.)

### DC-A. 타입별 필드 정합성 (개념 정리)
- **A1** 레포 타입은 프롬프트(body) 필드가 **없다**. 상세 뷰에 body/URL을 "프롬프트"처럼 표시하지 않는다(레포는 GitHub 열기 링크만). 편집 폼에도 프롬프트 입력칸 없음.
- **A2** "프롬프트" 라벨은 body를 가리키며 prompt/image_prompt 타입에만 나타난다. "메모"는 notes로 전 타입 공통. 둘을 혼동하는 표기 없음.
- **A3** body 라벨·placeholder가 전부 "프롬프트"로 통일("본문" 잔재 없음). DB 컬럼명 `body`는 불변(라벨만 변경).

### DC-B. 제목 처리
- **B1** 모달 내 제목은 **헤더 h2 한 곳에만** 존재. 본문 하단 중복 제목 섹션(`detail-title-section`) 부재.
- **B2** **detail 모드** 헤더는 2줄(뱃지+액션 / 제목), 제목 line-clamp:1. **편집/추가 모드** 헤더는 제목 h2 없음(뱃지+액션만) — 폼 입력칸이 제목 담당해 중복 제거. "편집"·"새 항목" 문구도 없음.
- **B3** 상세·편집·카드 모두 **제목이 최상단**. 모달만 제목이 아래에 있는 불일치 없음.

### DC-C. 순서 일관성 (카드와 통일)
- **C1 상세 순서**: 헤더 → 프롬프트(레포는 GitHub 링크) → (이미지) → 메모 → 태그 → 구분선 → [날짜·출처]. 콘텐츠(프롬프트/메모) 위, 참조(출처 링크)는 최하단 메타줄.
- **C2 편집 순서**(전 타입 통일): 제목 → 프롬프트/GitHub URL → (이미지 업로드, 프롬프트 바로 아래) → 메모 → 출처 링크 → 태그. 이미지 편집만 제목이 2번째인 불일치 없음.
- **C3** 메모가 출처 링크보다 위(콘텐츠 우선). 모든 타입 동일.

### DC-D. 표준 유지 (임의 UI 금지)
- **D1** 링크 복사를 위한 **별도 아이콘을 새로 만들지 않는다**. 헤더의 기존 복사 버튼 하나로 본문+GitHub URL 복사(표준 확장).
- **D2** 복사 버튼은 프롬프트·이미지·레포 상세 모두 표시(복사 대상 있으면). 출처 링크는 복사 대상 아님.

### DC-E. 표기·자산
- **E1** 수정일은 "YYYY-MM-DD 수정" 형식("수정일 " 접두 없음).
- **E2** 파비콘: index.html에 favicon.ico 참조 없음, favicon.svg만.
