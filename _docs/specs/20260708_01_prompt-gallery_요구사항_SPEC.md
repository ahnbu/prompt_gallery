# Prompt Gallery 요구사항 SPEC

## 목적
Prompt Gallery는 개인 작업에서 자주 쓰는 프롬프트, 이미지 프롬프트, Workflow, 레포를 빠르게 찾고 바로 실행할 수 있게 하는 작업 허브다.

단순 보관함이 아니라 다음 상황을 모두 지원한다.
- 작업 시작 전: 지금 할 일에 맞는 프롬프트, Workflow, 레포를 빠르게 찾는다.
- 아이디어 정리 중: 좋은 프롬프트와 자료를 부담 없이 모아둔다.
- 실행 도중: 필요한 프롬프트를 즉시 복사하고, 관련 Workflow나 GitHub 레포를 연다.

## 요구사항
- R1. 사용자는 프롬프트, 이미지 프롬프트, Workflow, 레포를 한 앱에서 검색하고 탐색할 수 있어야 한다.
- R2. 검색은 기본적으로 전체 항목을 대상으로 하되, 탭과 태그 필터로 좁힐 수 있어야 한다.
- R3. 태그 필터는 여러 태그를 선택했을 때 AND 조건으로 동작해야 한다.
- R4. 프롬프트와 이미지 프롬프트는 카드와 상세 모달에서 본문만 즉시 복사할 수 있어야 한다.
- R5. 새 항목 추가는 필수 입력을 최소화해야 한다. 프롬프트 본문은 필수이고, 제목과 태그는 선택이다.
- R6. 제목이 없는 프롬프트는 본문 앞부분을 제목처럼 보여줘야 한다.
- R7. 항목의 태그는 사용자가 직접 추가·삭제할 수 있고, 앱이 자동으로 붙인 태그도 함께 활용해야 한다.
- R8. 태그 관리 화면에서 전체 태그의 이름 수정, 삭제, 합치기, 자동 분류 기준 관리가 가능해야 한다.
- R9. Workflow는 순서가 있는 그루핑으로, 단계별로 프롬프트, 레포, 메모, 외부 링크를 담을 수 있어야 한다.
- R10. Workflow는 이름과 최소 1개 단계가 있어야 저장할 수 있어야 한다.
- R11. 레포 항목은 독립적으로 찾을 수 있고 Workflow 단계에도 연결될 수 있어야 한다.
- R12. 레포 항목의 1차 행동은 GitHub 온라인 저장소 페이지 열기여야 한다.
- R13. 이미지 프롬프트는 별도 탭으로 제공하고, 결과 이미지 preview 중심으로 탐색할 수 있어야 한다.
- R14. 이미지 프롬프트의 결과 이미지는 선택 입력이며, 이미지가 없을 때는 lucide 아이콘과 짧은 문구로 빈 상태를 표시해야 한다.
- R15. 카드 클릭 시 상세 모달에서 조회, 수정, 삭제가 가능해야 한다.
- R16. 삭제는 바로 실행하지 않고, 한 번 확인한 뒤 삭제해야 한다.
- R17. 추가·수정은 자동 저장이 아니라 명시적 저장 버튼으로 반영해야 한다.
- R18. 사용자는 카드 우측 상단의 별표 아이콘으로 항목을 즐겨찾기할 수 있고, 상단 즐겨찾기 탭에서 즐겨찾기 항목만 볼 수 있어야 한다.

## 성공 기준
- 사용자는 상단 통합검색, 탭, 태그 목록을 통해 원하는 항목을 찾을 수 있다.
- 사용자는 프롬프트 카드 또는 상세 모달에서 본문을 한 번에 복사할 수 있다.
- 사용자는 새 프롬프트를 본문만으로 저장할 수 있고, 저장 후 목록으로 돌아올 수 있다.
- 사용자는 항목 상세 모달에서 제목, 본문, 태그, 이미지, 메모를 수정하고 저장할 수 있다.
- 사용자는 Workflow를 이름과 최소 1단계로 만들고, 단계별 항목을 관리할 수 있다.
- 사용자는 레포 카드나 상세에서 GitHub 페이지를 열 수 있다.
- 사용자는 상단 관리 아이콘을 통해 태그 관리 화면에 들어갈 수 있다.
- 이미지가 없는 이미지 프롬프트 카드도 갤러리 레이아웃을 깨지 않고 구분된다.
- 사용자는 즐겨찾기 별표를 토글하고, 즐겨찾기 탭에서 항목 종류와 무관하게 즐겨찾기한 항목만 볼 수 있다.

## 제약 조건
- 사용자는 개발자가 아니므로 UI 문구와 관리 경험은 내부 구현 용어가 아니라 사용자 언어로 제공한다.
- 초기 목표는 개인용 도구다. 과도한 협업, 권한, 공개 배포 기능은 1차 범위가 아니다.
- 로컬 폴더 직접 열기는 브라우저 보안 제약이 크므로 1차 범위에서 제외한다.
- 이미지 preview는 초기에는 앱 안의 압축 이미지 파일 또는 정적 자산 중심으로 다루고, Cloudflare Images 같은 유료 이미지 파이프라인은 초기 기본값으로 두지 않는다.
- Google Drive는 보관 용도로는 가능하지만, 카드 preview의 안정적인 원천으로 전제하지 않는다.

## 결정안
### 정보 구조

- 1단 탐색: 통합검색 + 탭.
- 2단 탐색: 태그 리스팅과 AND 필터.
- 탭: `즐겨찾기`, `All`, `프롬프트`, `이미지 프롬프트`, `Workflow`, `레포`.
- `즐겨찾기` 탭은 항목 종류 탭이 아니라, 즐겨찾기한 항목만 보여주는 특수 필터 탭이다.
- 상단 탭은 `scrap_sns`처럼 compact 아이콘 버튼으로 표시한다.
- 탭의 full text label은 상시 노출하지 않고, hover tooltip과 접근성 라벨로 제공한다.
- 태그 필터 칩은 의미 전달이 핵심이므로 텍스트 칩 형태를 유지한다.
- `All` 기본 화면은 종류별 섹션으로 묶어 보여준다.
- 검색어 입력 시에는 섹션을 없애고 통합 카드 리스트로 보여준다.

트레이드오프: 기본 탐색은 구조가 중요하고, 검색 중에는 속도가 중요하다. 뱃지가 있으면 검색 결과를 섞어도 종류 구분이 가능하다.

### 카드와 액션

- 모든 항목은 공통 사각형 카드 스타일을 사용한다.
- 왼쪽 상단에 `프롬프트`, `이미지`, `Workflow`, `레포` 뱃지를 표시한다.
- 우측 상단에 즐겨찾기 별표 아이콘 버튼을 표시한다.
- 카드 색상만으로 종류를 구분하지 않고, 뱃지 텍스트를 함께 쓴다.
- 본문 미리보기는 고정 글자 수가 아니라 카드 영역 안에서 보이는 만큼만 표시한다.
- 태그는 기본적으로 모두 보여주되, 많을 때는 최대 10개까지만 노출한다.
- 프롬프트와 이미지 프롬프트의 핵심 액션은 `복사`다.
- Workflow의 핵심 액션은 상세 열기다.
- 레포의 핵심 액션은 `GitHub 열기`다.
- 기본 정렬은 최신순이다.

트레이드오프: 완전히 동일한 카드는 기능 차이를 흐리고, 종류별 전용 카드는 화면을 산만하게 만든다. 공통 카드와 종류별 핵심 액션의 조합이 균형적이다.

### 상세 모달과 관리

- 카드 클릭 시 상세 모달을 연다.
- 상세 모달에서 조회, 수정, 삭제가 가능하다.
- 삭제 버튼은 한 번 확인한 뒤 실행한다.
- 추가와 수정은 자동 저장이 아니라 명시적 저장으로 반영한다.

트레이드오프: 별도 관리 화면만 두면 흐름이 무거워지고, 카드만으로 끝내면 수정·삭제 위치가 불명확하다. 모달이 가장 직접적이다.

### 프롬프트 추가

- 상단 `+ 추가` 버튼을 제공한다.
- 현재 탭에 맞는 항목 추가 화면을 기본으로 연다.
- `All` 탭에서는 추가할 항목 유형을 먼저 고른다.
- 프롬프트 본문은 필수, 제목과 태그는 선택이다.
- 제목이 없으면 본문 앞부분을 제목처럼 표시한다.
- 저장 전 자동 태그 결과 확인 화면은 두지 않는다.

트레이드오프: 빠른 수집이 우선이므로 입력 부담을 줄인다. 자동 분류가 틀렸을 때는 상세에서 고치게 한다.

### 태그

- 자유 태그와 태그 카탈로그를 함께 쓴다.
- 태그 카탈로그의 역할은 검색 확장과 자동 태그 부여다.
- 항목 수정 화면에서 해당 항목의 태그를 추가·삭제할 수 있다.
- 상단 관리 아이콘에서 전체 태그 관리 화면으로 들어간다.
- 태그 관리 화면은 태그 이름 수정, 삭제, 합치기, 자동 분류 기준 관리를 맡는다.
- 자동 분류 기준은 태그 수정 화면 안에서 함께 편집한다.
- 태그 수정 화면에는 태그 이름, 색상, 자동 분류 키워드를 함께 둔다.
- 자동 분류 키워드는 사용자가 이해하기 쉽게 "이 단어가 나오면 이 태그를 붙임"의 의미로 표시한다.

트레이드오프: 태그 관리 화면은 범위를 키우지만, 태그가 핵심 탐색 축이므로 1차에 포함하는 편이 맞다.

### Workflow

- Workflow는 순서가 있는 그루핑이다.
- 예: `리서치 → 스토리 → 슬라이드계획 → PPT제작`.
- Workflow는 이름과 최소 1개 단계가 있어야 저장된다.
- 각 단계에는 프롬프트, 레포, 메모, 외부 링크를 넣을 수 있다.
- 기본은 순서대로 보이지만 단계별 접기, 검색, 바로 복사가 가능해야 한다.

트레이드오프: Workflow를 단순 컬렉션으로 만들면 원래 의도인 작업 순서 지원이 약해진다. 단, 처음부터 전체 단계를 강제하면 만들기 부담이 크다.

### 이미지 프롬프트

- 이미지 프롬프트는 별도 탭으로 제공한다.
- 결과 이미지 preview 중심의 갤러리형 탐색을 기본으로 한다.
- 용도, 스타일, 모델은 별도 메뉴가 아니라 태그로 처리한다.
- 결과 이미지는 선택 입력이다.
- 이미지가 없으면 lucide 아이콘과 짧은 `이미지 없음` 문구로 표시한다.
- 별도 placeholder 이미지 파일은 사용하지 않는다.

트레이드오프: 이미지 프롬프트는 결과물이 강한 탐색 단서다. 다만 이미지가 없어도 빠르게 저장할 수 있어야 하므로 preview는 선택으로 둔다.

## 사실 확인
- `scrap_sns`는 태그 카탈로그의 aliases를 검색 확장과 자동 태그 규칙으로 활용하는 패턴을 갖고 있다. URL별 태그 배열과 태그 카탈로그를 분리해 사용한다.
- `my_bookstation`은 사용자가 붙인 태그와 자동으로 붙인 태그를 구분하되, 표시·필터에서는 합쳐서 쓰는 패턴을 갖고 있다.
- `my_bookstation`의 설정 모달은 `태그관리` 탭 안에서 태그 목록, 사용량, 자동 키워드 개수, 수정, 삭제를 함께 보여준다. 태그별 `자동` 버튼을 누르면 `자동태그 키워드` textarea, `자동 적용` 체크박스, 저장/취소 버튼이 열린다.
- `my_bookstation`의 태그 정렬은 색상 우선, 사용량 많은 순을 사용한다. Prompt Gallery의 카드 정렬 기준과 동일하게 가져올 필요는 없지만, 태그 관리 화면 안에서는 참고할 수 있다.
- Cloudflare Pages 공식 문서 기준, Functions를 호출하지 않는 정적 자산 요청은 free/paid plan 모두에서 무료·무제한으로 설명된다. 참고: [Cloudflare Pages Functions pricing](https://developers.cloudflare.com/pages/functions/pricing/)
- Cloudflare R2 공식 문서 기준, egress 비용은 없고 저장량과 Class A/B operation 기준으로 과금된다. 참고: [Cloudflare R2 pricing](https://developers.cloudflare.com/r2/pricing/)
- Cloudflare Images 공식 문서 기준, 저장·전송·변환에 별도 과금 항목이 있다. 초기 개인용 preview에는 과한 선택이다. 참고: [Cloudflare Images pricing](https://developers.cloudflare.com/images/pricing/)
- Google Drive는 파일 보관·공유 도구로 설명되며, 앱 카드 preview의 안정적인 이미지 CDN으로 전제하기에는 적합성이 낮다. 참고: [Google Drive](https://workspace.google.com/products/drive/)

## 의사결정 로그
| 시점 | 결정 내용 | 근거 | 검토한 대안 |
|------|-----------|------|-------------|
| 탐색 구조 결정 | 통합검색 + 탭 + 태그 리스팅 | 검색과 둘러보기를 모두 지원해야 함 | 검색창 단독 |
| 태그 필터 결정 | 다중 태그는 AND 조건 | 선택한 모든 조건을 만족하는 항목을 찾는 사용 기대에 부합 | OR 조건 |
| 큰 범위 결정 | 프롬프트, 이미지 프롬프트, Workflow, 레포를 모두 1차 포함 | 사용자는 찾기, 정리, 실행을 모두 필요로 함 | 프롬프트만 1차 구현 |
| 태그 역할 결정 | 검색 확장과 자동 태그 부여 | `scrap_sns` 방식과 사용자의 의도에 부합 | 검색 확장만, 태그 추천까지 |
| 입력 흐름 결정 | 필수 입력 최소화, 자동 태그 확인 단계 없음 | 빠른 수집 흐름이 핵심 | 저장 전 자동 태그 검토 |
| Workflow 결정 | 순서가 있는 단계형 그루핑 | 기획서 작성 같은 작업 흐름을 따라가야 함 | 단순 컬렉션 |
| 이미지 프롬프트 결정 | 별도 탭과 preview 중심 카드 | 이미지 프롬프트는 결과물이 탐색 단서 | 용도별 탭, 모델별 탭 |
| 이미지 없음 결정 | lucide 아이콘 + 짧은 문구 | placeholder 이미지 로딩 없이 상태를 명확히 표시 | 기본 이미지 파일 |
| 복사 결정 | 프롬프트 본문만 복사 | 바로 붙여넣는 실행성이 가장 중요 | 제목·태그 포함 복사 |
| 상세 결정 | 카드 클릭 시 조회·수정·삭제 모달 | 추가 후 수정·삭제 위치가 필요 | 목록 카드만, 별도 관리 화면 |
| 삭제 결정 | 한 번 확인 후 삭제 | 실수 방지와 단순함의 균형 | 즉시 삭제, 삭제 후 되돌리기 |
| 저장 결정 | 명시적 저장 | 긴 프롬프트 수정에서 예측 가능성 확보 | 자동 저장 |
| 추가 버튼 결정 | 상단 추가 버튼 + 현재 탭 기본값 | 빠르면서도 유형 변경 가능 | 탭별 전용 버튼, 매번 유형 선택 |
| 레포 결정 | GitHub 열기만 1차 제공 | 브라우저에서 로컬 폴더 열기는 안정 기능으로 보기 어려움 | 로컬 폴더 열기, 경로 복사 |
| 태그 관리 결정 | 상단 관리 아이콘에서 별도 태그 관리 | 탐색 화면은 단순하게 유지 | 태그 영역 근처 관리 버튼 |
| All/검색 결과 결정 | All 기본은 섹션 구분, 검색 결과는 통합 리스트 | 둘러보기는 구조, 검색은 속도가 중요 | 항상 섹션 유지, 항상 통합 |
| 카드 표시 결정 | 본문은 카드 안에서 보이는 만큼만 표시 | 고정 글자 수보다 반응형 카드 레이아웃에 맞음 | 글자 수 고정 |
| 태그 노출 결정 | 카드 태그는 최대 10개 노출 | 기본은 다 보여주되 과밀한 카드만 제어 | 무제한 노출, 3~5개 제한 |
| 정렬·즐겨찾기 결정 | 기본 최신순, 별표 토글, 즐겨찾기 탭 제공 | 단순한 기본 정렬에 빠른 재접근 수단을 추가 | 자주 쓰는 순, 최근 복사순 |
| 자동 분류 UX 결정 | 태그 수정 화면에서 이름·색상·자동 키워드를 함께 편집 | 태그의 의미와 자동 분류 기준을 한 화면에서 이해하기 쉬움 | `my_bookstation`처럼 별도 `자동` 버튼, 자동 분류 전용 화면 |
| 탭 표시 방식 결정 | 상단 탭은 compact 아이콘 중심 버튼으로 표시 | 검색·필터 영역의 밀도를 유지하고 `scrap_sns` 화면 흐름과 맞음 | full text 탭 메뉴 |
