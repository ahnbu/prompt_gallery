# Prompt Gallery R2 비용폭주 방지 의사결정

## 1. 목적

Prompt Gallery 구현 전 Cloudflare R2, Workers, D1 사용에서 비용이 예상보다 커지는 상황을 막기 위한 1차 보호 수준을 결정한다.

이 문서는 구현계획 전체를 다시 쓰기 위한 문서가 아니다. 비용관리 판단을 구현계획, 백로그, 운영 체크리스트로 나누기 위한 기준만 고정한다.

## 2. 현재 확인된 상태

- R2 bucket `prompt-gallery-previews`는 생성 완료됐다.
- Cloudflare Dashboard 화면에서 `prompt-gallery-previews`의 공개 액세스는 `사용 안 함`으로 확인됐다.
- bucket 크기는 `0 B`로 확인됐다.
- `workers.dev` 주소에도 Cloudflare Access를 적용할 수 있다.
- 별도 custom domain은 1차 구현의 필수 조건이 아니다.
- Cloudflare에는 R2, Workers, D1 비용을 금액 기준으로 자동 중단하는 hard cap 기능이 없다.
- 현재 구현계획에는 이미 `Cloudflare Access 전제`, `R2 public bucket 금지`, `Worker proxy preview`, `원본 이미지 미보관`이 들어 있다.

## 3. 핵심 판단

비용폭주 방지의 핵심은 두 가지다.

1. 외부인이 앱을 사용하지 못하게 한다.
2. 사용자가 큰 이미지를 올려도 R2에 큰 원본이 그대로 저장되지 않게 한다.

Cloudflare Access는 1번을 담당한다. 이미지 압축과 업로드 제한은 2번을 담당한다. 다만 2번의 세부 스펙은 현재 구현계획에 끼워 넣지 않고, 별도 승인 후 백로그 항목으로 구현계획을 세운다.

## 4. 검토한 선택지

| 선택지 | 판단 |
|---|---|
| Budget Alert만 사용 | 알림일 뿐 실제 요청을 막지 못하므로 단독으로는 부족하다. 사용자가 Dashboard에서 별도 설정할 운영 항목이다. |
| R2 public access 차단 | R2 객체를 Worker API 없이 외부 URL로 직접 여는 기능을 쓰지 않는다는 뜻이다. 현재 이미 `사용 안 함` 상태라 새 구현 작업이 아니라 유지 사항이다. |
| 파일 3MB 제한 | 일반 스크린샷이나 생성 이미지에서 자주 걸릴 수 있어 사용성이 나쁘다. |
| 파일 10MB 제한 | 이례적으로 큰 파일만 막으면서 대부분의 사용 흐름은 유지된다. |
| 업로드 전 자동 압축/리사이즈 | 사용성 손상이 적고 R2 저장량을 줄이는 핵심 보호 장치다. |
| 이미지 조회 제한 | 카드 preview를 자주 보면 정상 사용도 막힐 수 있어 1차에서는 제외한다. |
| 월 업로드 개수/용량 제한 | 비용 방어에는 좋지만 1차 구현에서는 집계 복잡도가 늘어난다. 필요 시 나중에 추가한다. |
| KILL_SWITCH | 현재 단계에서는 과하다. Access와 R2 public bucket 금지만으로 1차 리스크를 줄이고, 실제 운영 중 필요성이 생기면 다시 검토한다. |
| custom domain + Access | 더 정돈된 운영 방식이지만 별도 설정이 필요하다. 아무것도 추가하지 않으면 기본 배포 대상은 `workers.dev`다. |
| workers.dev + Access | 기존 계획의 `Cloudflare Access 전제`만으로 충분하다. custom domain 필수 아님을 구현계획에 별도 보강할 필요는 낮다. |

## 5. 최종 선택안

1차 기준은 다음과 같이 나눈다.

현재 구현계획으로 충분한 항목:

- Cloudflare Access 보호 방향은 유지한다.
- Cloudflare Access는 앱 코드로 로그인 기능을 만드는 일이 아니라, 배포 후 Cloudflare Dashboard에서 접근을 허용 이메일 기준으로 막는 운영 설정이다.
- custom domain은 별도 설정이 필요한 선택지이며, 1차 필수 조건으로 두지 않는다.
- R2 bucket `prompt-gallery-previews`의 공개 액세스 `사용 안 함` 상태를 유지한다.
- R2에는 원본 이미지가 아니라 압축된 preview 파일만 저장한다.
- R2 public bucket 또는 public URL 기반 preview는 만들지 않는다.

별도 백로그로 보낼 항목:

- 사용자가 선택하는 원본 이미지 파일 10MB 제한.
- 업로드 전 브라우저 preview 자동 리사이즈/압축.
- 긴 변 1600px 이하, WebP 우선/JPEG fallback.

운영 체크리스트로 분리할 항목:

- Budget Alert는 Cloudflare Dashboard에서 사용자가 별도 설정한다.

## 6. 별도 백로그 항목

아래 항목은 현재 구현계획에 바로 넣지 않고, `BACKLOG.md`의 `BL-0708-01`에서 별도 승인 후 구현계획을 세운다.

이미지 업로드 비용 방어 스펙:

- 원본 파일이 10MB를 초과하면 선택 단계에서 안내하고 저장하지 않는다.
- 원본 파일이 10MB 이하이면 브라우저에서 자동 리사이즈/압축한다.
- 긴 변은 1600px 이하로 줄인다.
- WebP를 우선 사용하고, 브라우저 지원이나 변환 실패 시 JPEG를 fallback으로 사용한다.
- 압축된 preview 파일은 대체로 1MB 안팎을 기대한다. 단, 1MB는 목표값이지 사용자에게 노출하는 hard limit은 아니다.
- 모바일에서 변환이 실패하면 원본 업로드로 우회하지 않고 더 작은 이미지를 선택하라고 안내한다.
- R2에는 압축된 preview 파일과 필요한 metadata만 저장한다.

Budget Alert는 `BACKLOG.md`의 `BL-0708-02`에서 배포 전 운영 체크리스트로 분리한다.

## 7. 제외한 항목

1차 구현에서는 다음을 제외한다.

- 파일 3MB 제한.
- 이미지 조회 일일 제한.
- 월 업로드 개수 제한.
- 월 업로드 총량 제한.
- custom domain 필수화.
- 앱 자체 로그인 구현.
- R2 public bucket 또는 public URL 기반 preview.
- KILL_SWITCH.

제외 이유는 단순하다. 비용 방어 효과보다 구현 복잡도나 사용성 손상이 더 크기 때문이다. 이 항목들은 실제 사용량이 늘거나 외부 노출 리스크가 커졌을 때 다시 검토한다.

- `workers.dev`는 custom domain을 따로 붙이지 않으면 기본 배포 대상이므로 구현계획에 별도 보강하지 않는다.
- R2 공개 액세스 `사용 안 함`은 운영 상태 확인 항목이지 새 구현 작업으로 과장하지 않는다.
- 이미지 압축 세부 스펙과 Budget Alert는 `BACKLOG.md`로 분리한다.
- `KILL_SWITCH`는 현재 구현계획에 넣지 않는다.

## 9. 적용 기준

이 문서는 handoff가 아니라 비용관리 판단 문서다. 이후 구현계획을 수정할 때는 아래 기준을 따른다.

- R2 공개 설정은 이미 원하는 상태이므로 새 작업으로 과장하지 않는다.
- 3MB 제한은 사용성이 나빠서 채택하지 않는다.
- 조회 제한과 월간 사용량 집계는 1차에서 제외한다.
- 구현계획에는 이미 들어간 접근 보호와 R2 비공개 흐름을 유지한다.
- 이미지 업로드 비용 방어 세부 스펙은 별도 승인 후 `BL-0708-01`로 계획화한다.
- Budget Alert는 구현계획이 아니라 운영 체크리스트로 둔다.
- `KILL_SWITCH`는 현재 단계에서는 제외하고 필요 시 재검토한다.
