# 배포 migration 누락 방지 계획

## 목표

원격 D1에 적용되지 않은 migration이 있을 때 Worker 배포가 진행되지 않도록 배포 전 검사를 강화한다. 배포 실행은 `build -> remote migration apply -> deploy check -> wrangler deploy` 순서로 고정한다.

## 배경

2026-07-10 KST에 `/api/items` GET/POST가 500 Worker 1101로 실패했다. 원인은 원격 D1 `prompt-gallery-db`에 `0002_source_aware_item_tags.sql`이 적용되지 않아 현재 Worker 코드가 참조하는 `item_tags.source` 컬럼이 없던 것이다.

## 범위

### 포함

- `scripts/qa/deploy-check.mjs`에 로컬 migration 파일과 원격 `d1_migrations` 적용 목록 비교 추가
- `package.json`에 안전 배포 스크립트 추가
- 배포 검사 로직의 실패/성공 테스트 추가

### 제외

- DB 스키마 자체 변경
- 원격 D1 데이터 수정
- Cloudflare Worker 라우팅 변경
- 자동 rollback 구현

## 파일 구조

- 수정: `scripts/qa/deploy-check.mjs`
  - 책임: 배포 전 로컬 설정, 빌드 산출물, 원격 D1/R2, 원격 D1 migration 적용 상태를 검증한다.
- 생성: `scripts/qa/deploy-check.test.mjs`
  - 책임: Wrangler 실행을 mock runner로 대체해 pending migration 실패와 current migration 성공을 검증한다.
- 수정: `package.json`
  - 책임: 사람이 기억해야 하는 배포 순서를 `deploy:prod` 스크립트로 고정한다.

## 작업 계획

### Task 1. `deploy-check`를 테스트 가능하게 분리한다

완료 기준:

- `deploy-check.mjs`가 CLI 직접 실행 시 기존처럼 동작한다.
- 테스트에서는 `runDeployCheck({ runner, rootDir, outputPath })` 같은 진입점으로 Wrangler 응답을 주입할 수 있다.
- CLI 전용 `main()`은 파일 하단에서만 호출된다.

검증 방법:

- `node scripts/qa/deploy-check.mjs --output .omo/evidence/deploy-check-plan-smoke.txt`
- 기대 결과: exit code 0, 출력에 `PASS deploy check` 포함

### Task 2. 원격 D1 migration 누락 검사를 추가한다

완료 기준:

- 로컬 `migrations/*.sql` 파일명 목록을 정렬해 읽는다.
- 원격 D1은 `wrangler d1 execute prompt-gallery-db --remote --command "SELECT name FROM d1_migrations ORDER BY id"`로 조회한다.
- 로컬에는 있지만 원격에 없는 migration이 있으면 `DeployCheckError`로 실패한다.
- evidence 파일 `## Checks`에 성공 시 `remote D1 migrations current`가 남고, 실패 시 `Pending remote D1 migrations: ...`가 남는다.

검증 방법:

- `node --test scripts/qa/deploy-check.test.mjs`
- 기대 결과: pending migration fixture는 실패를 검증하고, current migration fixture는 통과를 검증한다.

### Task 3. 안전 배포 스크립트를 추가한다

완료 기준:

- `package.json`에 아래 스크립트가 추가된다.

```json
"deploy:prod": "pnpm build && pnpm exec wrangler d1 migrations apply prompt-gallery-db --remote && pnpm deploy:check && pnpm exec wrangler deploy"
```

검증 방법:

- `pnpm run`
- 기대 결과: script 목록에 `deploy:prod`가 포함된다.

### Task 4. 전체 검증을 수행한다

완료 기준:

- 테스트, 타입체크, lint, 빌드, 배포 체크가 모두 통과한다.
- 원격 D1 현재 migration 목록이 로컬 목록과 일치한다.
- 라이브 `/api/items` 응답이 200이다.

검증 명령:

```bash
node --test scripts/qa/deploy-check.test.mjs
pnpm typecheck
pnpm lint
pnpm test:worker
pnpm build
pnpm deploy:check
pnpm exec wrangler d1 execute prompt-gallery-db --remote --command "SELECT name FROM d1_migrations ORDER BY id"
```

기대 결과:

- `node --test`는 모든 deploy-check 테스트 통과
- `pnpm typecheck`, `pnpm lint`, `pnpm test:worker`, `pnpm build`, `pnpm deploy:check` exit code 0
- 원격 migration 목록에 `0001_initial.sql`, `0002_source_aware_item_tags.sql` 포함

## 커밋 전략

- 한 커밋으로 묶는다.
- 메시지 후보: `fix(deploy): 원격 D1 migration 누락 검사 추가 — 배포 전 스키마 불일치 차단`

## 리스크와 대응

- Wrangler 출력 포맷이 바뀔 수 있다.
  - 대응: JSON 파싱을 우선하고 실패 시 안전하게 실패 메시지를 남긴다.
- `deploy-check.mjs`가 커질 수 있다.
  - 대응: helper 함수 단위로 책임을 나누되, 별도 파일 분리는 이번 범위에서 제외한다.
- `deploy:prod` 실행 중 migration apply가 이미 적용된 상태일 수 있다.
  - 대응: Wrangler migration apply의 no-op 동작을 사용하고, 이후 `deploy:check`가 최종 상태를 검증한다.

## 자체 검수

- 목적 적합성: 이번 장애 원인인 원격 migration 누락을 배포 전 차단한다.
- 범위 통제: 코드 배포 안전장치만 다루며 DB 스키마와 데이터는 바꾸지 않는다.
- 검증 가능성: mock runner 기반 실패 테스트와 실제 원격 read-only 체크가 모두 포함되어 있다.
