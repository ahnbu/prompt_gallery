이 Plan 문서를 면밀히 검수한 결과, **UI 개선이라는 시각적 목적은 달성할 수 있으나, 미결정된 기획과 검증 기준 누락으로 인해 AI(또는 개발자)가 중단 없이 자율 실행하는 것은 불가능합니다.**

실행을 위해 문서를 전면 수정해야 합니다. 아래 검수 결과를 바탕으로 Plan을 보강할 것을 권장합니다.

### 1. 실행 불가 및 인간 개입(판단) 요구 항목
현재 계획에는 에이전트가 독자적으로 결정할 수 없는 열린 질문(Placeholder)이 포함되어 있어, 실행 중단(Block)이 발생합니다. 실행 전에 아래 항목들의 의사결정이 선행되거나 명확한 지시로 수정되어야 합니다.

| 위치 | 문제의 문구 | 리스크 및 수정 방향 |
| :--- | :--- | :--- |
| **변경계획 2번** | "모델 확인이 필요하다. 없다면 데이터/API 설계를 확인한다." | 실행 지시가 아님. **[사전 확인]** 단계로 분리하여 `WorkflowItem`의 DB 스키마/타입을 먼저 조회한 후, 복사 대상이 없으면 해당 기능을 숨길지 별도 필드를 만들지 확정해야 함. |
| **변경계획 3번** | "별도 판단이 필요하다." | 에이전트가 UI 임의 배치를 시도하거나 질문하며 멈추게 됨. "카드 하단에 남긴다" 또는 "상세 모달로 이동시킨다"로 **명확히 확정**해야 함. |

### 2. 완료 기준 및 검증 방법 누락
이 문서는 '무엇을 바꿀지'에 대한 방향(Direction)만 있고, 작업이 성공했는지 증명할 '완료 및 검증 기준(Acceptance Criteria)'이 전혀 없습니다. `done-check`가 증거 기반으로 판단할 수 없습니다.

- **누락된 검증 지표:**
  - 모달을 열었을 때 구조가 설계대로 렌더링되는지 확인하는 DOM/시각 지표 (예: `ItemModal 헤더에 제목 텍스트가 없고 Badge만 존재하는지 테스트`).
  - 기존의 수정(Edit) -> API Submit 흐름이 깨지지 않았는지 확인하는 기능 검증.
  - 모바일 해상도(CSS) 적용 결과 확인 방법.
- **조치안:** 각 변경계획 하위에 `[완료 기준]`과 `[검증 방법(로컬 실행 or 테스트 코드)]` 섹션을 필수적으로 추가해야 합니다.

### 3. 영구 데이터 및 API 페이로드 리스크
- **변경계획 7번(태그 칩 편집 반영):** 기존의 `tagsText`(쉼표 구분 문자열) 단일 input을 칩 리스트 배열(Array) UI로 변경합니다.
- **리스크:** UI 상태 관리가 `string`에서 `string[]`로 바뀔 가능성이 큽니다. 만약 폼 Submit 시 백엔드로 보내는 API 페이로드 타입까지 변경된다면, 백엔드 검증 로직이나 DB 저장 방식(영구 데이터)에 영향을 미칩니다.
- **조치안:** 기존 API 호환성을 유지하기 위해 UI(배열) ↔ API(쉼표 문자열) 간의 직렬화/역직렬화(Serialization) 로직 처리에 대한 명시와, 데이터 저장 성공 여부를 검증하는 단계가 포함되어야 합니다.

### 4. 설계 구조적 충돌점
- **수정 모달의 우측 상단 액션:** (변경계획 4번, 6번) 상세 모달과 수정 모달의 구조를 통일하면서 수정 모달에도 상단 우측에 `복사, 즐겨찾기` 아이콘을 배치하려 합니다.
- **충돌:** 일반적으로 '수정 상태(Edit Mode)'에서는 데이터가 변경 중이므로 '복사'나 '즐겨찾기' 상태 업데이트가 논리적으로 충돌하거나 혼란을 줄 수 있습니다(저장되지 않은 텍스트를 복사할 것인가, 원본을 복사할 것인가).
- **권장안:** 수정 모달의 상단은 `유형 배지`와 `닫기(X)`만 남기고, 복사/즐겨찾기는 상세(Read) 모달에서만 노출하는 것이 덜 복잡합니다.

### 5. 접근법의 트레이드오프
| 접근법 | 얻는 것 (Pros) | 잃는 것 (Cons) / 리스크 |
| :--- | :--- | :--- |
| **상세/수정 공통 레이아웃 통합** (5번) | 코드 중복(DRY) 감소, 시각적 일관성 확보 | 읽기 모드와 쓰기 모드의 조건부 렌더링(`if isEditing`)이 얽혀 `ItemModal` 컴포넌트의 복잡도가 크게 증가함. |
| **텍스트 버튼 → 우측 상단 아이콘 이동** (4번) | 하단 공간 확보, 화면이 깔끔해짐 | 기능의 발견성(Discoverability) 저하. 모바일 환경에서 상단 우측 터치 영역 확보 필요. |
| **제목을 본문 하단으로 이동** (5번) | 내용(본문)에 대한 즉각적인 몰입도 상승 | 일반적인 UI 관습(제목 -> 본문)에 위배되어 사용자가 정보의 구조를 파악하는 데 일시적 혼란을 겪을 수 있음. |

### 6. 검증 과정 산출물 처리 및 의존성
- **산출물/커밋 정책 누락:** 작업이 성공적으로 끝났을 때 변경사항을 커밋으로 바로 반영(`cp` 스킬)할지, 리뷰용으로 작업만 해둘지(Local-only) 의도가 닫혀있지 않습니다.
- **숨은 선후관계 (의존성):** CSS 파일 업데이트(8번)와 React 컴포넌트 구조 변경(1~7번)이 같은 단계로 나열되어 있습니다. 컴포넌트 DOM 트리가 확정(1~7번)된 이후에 CSS(8번)를 조정해야 하는 강한 선후관계가 존재합니다.
