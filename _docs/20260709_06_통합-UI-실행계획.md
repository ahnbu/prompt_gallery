# 통합 UI 실행계획 — 카드 · 상세/수정 모달 · 헤더 · 태그관리

## 결론

`_03` 변경계획(구조 재배치)과 v3 목업(비주얼 완성도 + 헤더/태그관리 확장)을 하나의 실행계획으로 통합한다. 작업을 **5개 웨이브**로 분리한다. `W1`(디자인 토큰 + DESIGN.md)이 선행이고, `W2~W5`는 서로 독립이라 병렬 실행 가능하다. 각 웨이브는 한 관심사 = 한 커밋 단위다.

- **정본 목업**: [[20260709_05_전체-UI-개선목업-v3]] (브라우저로 열면 hover·복사 토스트·즐겨찾기·태그삭제가 실제 동작한다.)
- `_03`의 변경계획 1~7번, 확정 UX 방향, 논의 이력은 이 문서의 웨이브에 그대로 흡수했다. `_03`은 폐기하지 않고 `supersedes` 관계로 남긴다.

## 통합의 근거 (충돌·중복 검증 완료)

v3 목업은 `_03`의 구조를 그대로 따르는 **상위집합**이며 충돌하지 않는다. 세션에서 목업 원문·현재 코드와 대조해 확인했다.

| 관계 | 내용 |
| --- | --- |
| `_03` 1~7번 | v3가 구조를 100% 계승 → 재사용 |
| `_03` 8번(CSS 검수) | v3가 비주얼 완성도 6종으로 **구체화·확장** |

| `BL-0709-01` | v3도 건드리지 않음 → 백로그에 그대로 유효 |

## 반영한 리뷰 피드백 (Codex 검증)

`_03` 세션(Codex)의 프롬프트 검증에서 아래 3건이 확인되어 이 계획에 반영했다.

1. **자동제목 중복 완화는 폰트 위계와 같은 메커니즘** → 별도 항목을 만들지 않고 W3의 제목/본문 위계 항목에 통합한다.
2. **탭 유형색 dot은 "선택 사항"이 아니라 전체 탭에 상시 구현** → W2에 "전체 탭 유형색 dot, 활성탭은 배경·테두리 강조"로 확정 기술한다.
3. **v2 목업 참조는 v3 정본과 모순** → 참조에는 v3(`_05`)만 링크한다.

## 반영한 검수 피드백 (agy plan-check, Gemini 3.1 Pro)

독립 reviewer(agy)가 실제 코드와 대조해 검수한 결과 중 아래를 반영했다.

1. **[크리티컬] W1 선행이 파일 충돌로 자동 강제되지 않음** → 웨이브 구조 섹션에 수동 통제 경고(별도 선행 wave로 W1 먼저 완료)를 추가했다.
2. **[크리티컬] 카드 인라인 태그 편집은 순수 UI가 아니라 저장 API를 건드리는 영구 데이터 변경** → W3에 콜백 체인(`onTagsChange`)·`updateItem` API·refetch 로직·영구 데이터 영향을 명시했다. (사용자 결정: 카드 인라인 편집 유지 + 로직 보강)
3. **[개선] hover·토스트는 grep으로 검증 불가** → W3·W4 검증을 코드 관점(`:hover` 선택자 속성, 토스트 렌더 요소 존재)으로 구체화했다.
4. **[개선-미반영] 웨이브별 push가 프로덕션에 미완성 UI 배포 위험** → **반영하지 않음.** 이 레포는 CI 자동배포가 없고(`.github/workflows/` 부재, `deploy` 스크립트 없음, `wrangler deploy` 수동), 1인 사용 서비스라 미완성 UI 노출 리스크가 실질적으로 없다. agy가 실제 배포 파이프라인을 확인하지 못하고 일반론으로 추정한 항목이다. 기존 방침(웨이브별 분리 커밋 + cp)을 유지한다.

## 웨이브 구조

```text
W1 (선행) ──┬── W2 헤더·탭·태그필터
            ├── W3 카드
            ├── W4 모달
            └── W5 태그관리 모달
```

W1 완료 후 W2~W5는 의존성이 없어 병렬 실행 가능하다. 단, 셋 이상을 동시에 스폰할 때는 공유 파일 충돌을 피한다(각 웨이브가 건드리는 CSS/컴포넌트가 분리되어 있어 충돌 없음).

> [!danger] W1 선행은 이 계획 문서 하나로는 `exec-plan`에 전달할 수 없다 — 실측으로 확인된 도구 한계
> `exec-plan`(`classify-waves.mjs`)의 알고리즘은 "파일이 겹치는 두 Task는 무조건 다른 wave"라는 이진 규칙이다. "1개 선행 + 나머지 4개는 그 뒤에 서로 병렬"이라는 2단 구조를 이 규칙만으로는 표현할 수 없다. 두 가지를 실측했다.
> - **대상 파일에 `base.css`를 안 넣음**: Task 1(W1)이 Task 2·4와 같은 Wave 1로 병렬 배치됨 → 선행 보장 실패.
> - **Task 2~5의 대상에 `base.css`를 추가**: Task 1~5가 전부 별도 Wave로 갈라져 완전 순차가 됨(Wave 1~5) → 선행은 되지만 Task 2~5의 병렬 이점이 사라짐.
>
> 즉 이 알고리즘 위에서 "1개 먼저 + 4개 병렬"은 **한 plan 파일로 표현 불가능**하다. 실행 시 아래처럼 물리적으로 2개 plan 파일로 나눠야 한다.
>
> **확정된 실행 절차**
> 1. Task 1(W1) 섹션만 담은 plan 파일을 만들어 `exec-plan`으로 단독 실행하고 완료를 확인한다.
> 2. Task 2~5(W2~W5) 섹션만 담은 plan 파일을 만들어 `exec-plan --review`로 실행한다. Task 2~5끼리는 파일이 겹치지 않아 정상적으로 병렬 처리된다.
> 3. 두 plan 파일은 이 통합계획의 해당 Task 섹션을 그대로 복사해 생성한다(내용 변경 없음, 분리만). 이 문서(`_06`)가 정본이고, 분리 파일은 실행용 파생본이다.

### Task 3: W3 — 카드

**대상**: `src/client/GalleryCard.tsx`, `src/client/GalleryList.tsx`, `src/client/App.tsx`, `src/client/styles/cards.css`

- 구조 순서: `배지+아이콘액션 → 본문 preview → 제목 → 태그`로 바꾼다(현재 제목→본문 순서 반전).
- **제목/본문 위계**: 제목 15.5px/700/primary, 본문 preview 12.5px/muted. 이 위계가 자동제목(제목=본문 첫 줄)일 때의 중복 체감을 낮추는 역할을 겸한다 — 별도 숨김 로직 없이 위계로 해결한다. (Codex 피드백 1 반영: "둘 다 표시" 유지)
- 본문 미디어 `align-content: start`(상단 정렬)로 붕 뜨는 문제 해소.
- 배지: 유형색 + 도트 마커. 워크플로우 배지 영문 `Workflow` → 한글 `워크플로우`.
- 카드 날짜(footer `updatedAt`) 제거.
- 복사: 텍스트 `복사됨` → 아이콘 체크 전환 + 토스트.
- hover: shadow lift + `translateY(-2px)`.
- 빈 상태(empty-state) 카드 스타일.
- 워크플로우 카드는 즐겨찾기 없이 스텝복사 액션만 유지(변경 없음 확인).

- **카드 태그 인라인 편집 (영구 데이터 변경 — 로직 보강 필수)**
  - 카드 태그를 칩 단위 삭제(×) + `+ Tag`로 편집 가능하게 한다. auto 태그는 삭제 버튼 없는 읽기전용 칩.
  - **주의**: 현재 `GalleryCard`는 순수 뷰 컴포넌트로 태그 변경 콜백이 없다(`GalleryList.tsx:11`의 props에 `onFavoriteChange`/`onOpenItem`/`onOpenWorkflow`만 존재). 카드에서 태그를 즉시 편집하려면 저장 API까지 이어지는 데이터 흐름을 새로 뚫어야 한다. 이는 `favorite` 토글이 이미 쓰는 패턴(`App.tsx:111 toggleFavorite`)과 동일한 구조로 구현한다.
  - 구현 항목:
    1. `GalleryCard`(및 `ItemCard`) props에 `onTagsChange(item, nextTags)` 추가.
    2. `GalleryList.tsx`의 `GalleryResults`→`SectionedList`/`UnifiedList`→`GallerySection`→`CardGrid`→`GalleryCard` 콜백 체인에 `onTagsChange`를 전파한다.
    3. `App.tsx`에 `changeTags(item, nextTags)` 핸들러를 추가한다. `updateItem(item.id, { tags: nextTags })`(`item-mutations.ts:58`, `PatchItemRequest`) 호출 후 `refreshGalleryData()`로 갱신한다. `toggleFavorite`와 동일한 패턴.
    4. auto 태그는 편집에서 제외한다(manual source만 변경 대상). 저장 payload 구성 시 auto 태그가 유실되지 않도록 기존 `visibleTags`/source 구분 로직을 따른다.
  - **영구 데이터 영향**: 이 편집은 `updateItem` API를 통해 아이템의 `tags`를 실제로 갱신한다. 기존 저장 스키마(`tags: string[]` 또는 tag source 구조)와 호환되는지 확인하고, auto 태그 재계산 규칙과 충돌하지 않는지 검증한다. 새 마이그레이션은 불필요하다(기존 필드 갱신).

**완료 기준**
- 카드 DOM 순서가 배지 → 본문 → 제목 → 태그다.
- 카드 footer에 날짜가 없다.
- 워크플로우 배지 텍스트가 `워크플로우`다.
- auto 태그 칩에 삭제 버튼(×)이 없고, 수동 태그 칩에는 있다.
- 카드에서 수동 태그를 삭제/추가하면 `updateItem` API가 호출되고, 새로고침 후에도 변경이 유지된다(영구 반영).
