### 3-2. All 탭 = Masonry 통합 뷰 (H) — 6번·3번 흡수
- **파일**: `gallery-model.ts` `shouldShowUnifiedResults`(137), `GalleryList.tsx` `UnifiedList`·`CardGrid`(59·191), `styles/cards.css`
- **변경**:
  1. All 탭에서도 통합 뷰 표시 (섹션 분할 제거). `shouldShowUnifiedResults`가 All을 포함하도록 조정하거나 UnifiedList를 All 기본으로
  2. **masonry 구현 = JS flexbox 컬럼 분배 (확정, 검수 §5 반영)**. `column-count`(순서 꼬임)와 CSS 네이티브 `grid-template-rows: masonry`(브라우저 지원 불완전) **모두 폐기**. → **scrap_sns 검증 패턴 재사용** (`D:/vibe-coding/scrap_sns/web_viewer/script.js:2340` `buildMasonryColumns`/`appendCards`).
     - 방식: 뷰포트 폭으로 컬럼 수 결정(예: ≥1536=4, ≥1024=3, ≥768=2, else 1) → 각 컬럼은 flexbox `flex-col` → 카드를 **"카드 개수가 가장 적은 컬럼"에 순서대로 분배**(scrap_sns는 `childElementCount` 최소 컬럼 선택).
     - **React 이식**: scrap_sns는 바닐라 JS DOM append. prompt-gallery는 React이므로 **직접 DOM 조작이 아니라** entries를 컬럼별 배열로 분배하는 순수 함수(예: `distributeIntoColumns(entries, colCount)`)를 만들고, 컬럼 배열을 렌더. resize 시 colCount 재계산(리스너 또는 CSS 미디어 + 상태).
  3. **정사각 강제 제거**(3번): `.gallery-card:not([data-card-type="image_prompt"])`의 `aspect-ratio: 1/1`(cards.css:26-29) 삭제, masonry에선 내용 높이대로
  4. **순서 트레이드오프 (명시)**: 컬럼 분배는 "최소 컬럼 채우기"라 **엄밀한 좌→우 순서가 아니라 높이/개수 균형 우선**. 최신 카드는 앞쪽에 오되 열 위치는 균형에 따름. scrap_sns가 감수하는 표준 트레이드오프이며, 완벽한 좌→우 + 밀도 동시 만족 CSS는 부재. **Tab 순서는 분배 배열 순서를 따르므로 시각 흐름과 크게 어긋나지 않음**(column-count의 세로 역행 문제는 없음).
- **완료기준**: All 탭에서 모든 타입 카드가 카테고리 구분 없이 한 masonry에, 빈 섹션 패널 없음, 정사각 여백 없음, 컬럼이 폭에 따라 1~4개로 반응, 카드 높이 내용대로
- **검증**: qa:browser에서 All 탭 렌더 → (1) 빈 섹션 DOM 부재, (2) 카드 높이가 내용에 따라 가변, (3) 뷰포트 폭별 컬럼 수 확인(1280→3열 등). 데스크톱·모바일 증거 스크린샷 `.omo/evidence/`
- **구현 참고**: 분배는 round-robin(`index % columnCount`)으로 순서 보존 — scrap_sns의 "최단 컬럼 채우기"는 렌더 후 높이 측정이 필요해 React 렌더 전 단계에 부적합. round-robin이 좌→우 순서·Tab 순서를 정확히 보존.
- **단위 테스트 참고**: 클라이언트 vitest 설정이 현재 레포에 없음(`pnpm test`는 worker 전용). `distributeIntoColumns`·`columnCountForWidth`는 순수 함수이나 클라이언트 테스트 인프라 부재로 qa:browser 통합 검증으로 대체. 인프라 신설은 A 범위 밖(별도 결정 사항).

## 배포·검증 (A 전체 완료 기준)

**검증은 전부 headless·증거 기반** (검수 §1 반영). 수동 육안 검증에 의존하지 않는다.

- `pnpm typecheck` · `pnpm lint` 통과 (종료코드 0)
- `pnpm test:worker` 통과 (API 계약 불변 — A는 클라이언트만 변경)
- `pnpm qa:browser` 통과 — **기존 Playwright 인프라는 이미 headless**(`scripts/qa/browser-*.mjs`의 `chromium.launch()`는 기본 headless=true, 창이 뜨지 않음). 각 시나리오의 `render*Evidence`가 증거를 남김.
- Wave1~3 검증용 시나리오 증거(스크린샷·stdout/stderr 로그)를 **`.omo/evidence/`**(gitignore됨)에 저장. done-check는 이 증거 파일과 종료코드로 판정.
- `pnpm build` 성공(종료코드 0) 후 배포. (deploy는 `pnpm exec wrangler deploy`, migration 불필요)

**검증 인프라 참고**: `startLocalApp(outputPath)`(browser-smoke-support.mjs:46)가 로컬 앱 기동 + stdout/stderr를 `{stem}.stdout/.stderr`로 파일 저장. 신규 UI 검증 시나리오가 필요하면 기존 `browser-*.mjs` 패턴을 따라 추가.
