## 관심사 3 — 이미지 업로드 UX (커밋 3)

리스크 중간 · 클라이언트 상태·이벤트 핸들러 추가. worker·R2 API는 변경 없음(기존 배선 재사용).

### 3-1. 저장 전 업로드 허용 (비활성 정책 완화)
- **파일**: `src/client/ImagePreviewField.tsx`
- **변경**:
  1. 파일선택 `disabled`(116행): `props.item === null || status !== "idle"` → **`status !== "idle"`** (item null 조건 제거)
  2. `canUpload`(25행): `selectedFile !== null && status === "idle" && props.item !== null` → **`props.item !== null` 제거**. 단, **draft 모드(`onDraftChange !== undefined`)일 때만 저장 전 업로드 허용**. draft 모드가 아니면서 item이 null인 경우는 없음(신규 추가는 항상 draft) → 조건: `selectedFile !== null && status === "idle" && (props.item !== null || props.onDraftChange !== undefined)`
  3. `uploadSelected`(31-34행) 가드: `props.item === null && props.onDraftChange === undefined`일 때만 early return. draft 모드면 itemId 없이 `uploadImageAsset({ file })` 호출(이미 42-44행에서 분기됨).
  4. `disabledReason`(26-29행): 안내문("항목 저장 후…")을 draft 모드에서는 표시 안 함(저장 전 업로드 가능해졌으므로)
- **완료기준**: 신규 이미지 프롬프트 추가 모달에서 저장 전에 파일선택·업로드가 활성, 업로드하면 draft에 assetId 반영, 항목 저장 시 연결
- **검증**: qa:browser(headless)에서 신규 이미지 프롬프트 모달 열고 → 파일선택 input `disabled=false` assertion → 파일 주입·업로드 → 프리뷰 img 표시 assertion → 저장 후 카드에 이미지 표시 assertion. `/api/assets` POST가 itemId 없이 발생하는지 네트워크 로그 확인. 증거 `.omo/evidence/`

### 3-1b. orphan 에셋 cleanup — add 케이스 대응 (필수, 검수 §2 반영)
- **배경(코드 확인)**: 모달 닫기 경로에 cleanup이 **이미 존재**한다 — [ItemModal.tsx:152,314-330](../src/client/ItemModal.tsx) `closeModal`이 `cleanupCurrentStagedAsset(state, draft)`를 호출해 staged 에셋(`draft.imageAssetId !== state.item.imageAssetId`)을 `deleteImageAsset`로 삭제. **그러나** 이 함수는 `state.item.imageAssetId`(321행)를 참조하는데, 신규 항목은 `ItemModalState = {kind:"add"}`([item-modal-model.ts:5-7](../src/client/item-modal-model.ts))라 **`state.item`이 없다**. 3-1로 저장 전 업로드를 새로 허용하면, **신규(add) 항목에서 이미지를 올린 뒤 취소 시 `state.item` 부재로 cleanup이 깨지고 orphan 에셋이 R2에 남는다**.
- **파일**: `src/client/ItemModal.tsx` `cleanupCurrentStagedAsset`(314행)
- **변경(확정, 검수 §2 반영 — "확인"이 아니라 반드시 구현)**: `cleanupCurrentStagedAsset`가 **add 케이스를 처리**하도록 수정. `state.kind === "add"`이면 `draft.imageAssetId`가 존재하는 모든 경우(비교 대상 원본 없음)를 staged로 간주해 `deleteImageAsset` 실행. `state.kind === "detail"`이면 기존 로직(`draft.imageAssetId !== state.item.imageAssetId`일 때만 삭제) 유지.
- **완료기준**: 신규 이미지 프롬프트에서 이미지 업로드 후 저장하지 않고 모달을 닫으면 해당 에셋이 삭제됨(orphan 미발생). 기존 항목 편집 취소 시 원본 에셋은 보존.
- **검증**: qa:browser(headless)에서 (1) 신규 이미지 프롬프트 모달 → 업로드 → 닫기 → `/api/assets` DELETE 발생 확인(네트워크 로그) + 해당 assetId 조회 시 404 assertion, (2) 기존 이미지 항목 편집 취소 시 원본 assetId가 살아있음 assertion. 증거 `.omo/evidence/`

### 3-2. 드래그앤드롭
- **파일**: `src/client/ImagePreviewField.tsx`, `src/client/styles/image-preview.css`
- **변경**:
  1. `image-preview-frame`(94행)에 `onDragOver`(preventDefault + dragging 상태 on), `onDragLeave`(off), `onDrop`(파일 추출 → `setSelectedFile` → 즉시 업로드 또는 파일선택과 동일 흐름) 핸들러 추가
  2. drop된 파일이 accept 타입(png/jpeg/webp)인지 검증, 아니면 error 표시
  3. dragging 상태일 때 프레임 테두리 강조 CSS(`image-preview.css`에 `.image-preview-frame.dragging` 추가)
  4. readOnly·compact 모드에서는 drop 비활성(상세 편집 폼에서만)
- **완료기준**: 이미지 프레임에 파일을 드롭하면 파일선택과 동일하게 업로드 흐름 진입. 드래그 중 시각 피드백
- **검증**: qa:browser(headless)에서 DataTransfer로 drop 이벤트 디스패치 → selectedFile 반영·업로드 진입 assertion. 잘못된 타입 drop 시 error 표시 assertion. 증거 `.omo/evidence/`
- **트레이드오프**: Playwright headless에서 native drag는 제한적 → `dispatchEvent`로 drop 이벤트 합성해 핸들러 로직을 검증(브라우저 native DnD UX 자체는 수동 확인 보조). 이 한계를 검증 항목에 명시.

## 부록 — 검토 근거

- **왜 제목을 위로?** 본문 길이 가변(1~3줄) → 제목이 아래면 카드마다 제목 y축이 달라 스캔이 어수선함(사용자 육안 지적). 제목 1줄 고정을 최상단에 두면 모든 카드 제목이 정렬됨. 이미지 카드 ②안(제목→이미지→본문)은 타입 카드와 제목 정렬이 일치(미리보기로 비교 후 사용자 선택).
- **왜 저장 전 업로드 허용?** `uploadImageAsset`이 itemId optional이라 배선상 이미 가능한데 UI만 막고 있었음. draft 모드가 이 경로를 쓰므로 정책만 완화하면 됨. "저장 후에만" 제약은 사용자 흐름을 끊음(항목 저장 → 다시 열기 → 이미지 추가).
- **왜 cursor:wait이 버그?** 비활성 컨트롤의 표준 커서는 `not-allowed`("지금 불가"). `wait`("시스템 처리 중")는 아무 작업이 없는데도 로딩 중처럼 오인시킴. 전역 수정이나 로딩 표현은 별도 `upload-progress`가 담당해 회귀 없음.
