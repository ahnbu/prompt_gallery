# 반응성(optimistic update) 전수 진단 — Wave 2 확장

> 진단일 2026-07-19 · 대상 src/client 전체 mutation 흐름 · 코드 전수 조사 기반

사용자 요청: "즐겨찾기 하나만이 아니라, 매번 느리다고 느낄 모든 지점을 한 번에 진단·개선하라."
결론부터: **즐겨찾기는 증상이고, 원인은 앱 전체가 공유하는 단일 안티패턴이다.**

---

## 1. 근본 원인 — 모든 쓰기가 같은 흐름을 탄다

모든 변경 동작이 예외 없이 이 경로를 따릅니다 (`App.tsx:75-114`):

```
사용자 클릭 → 서버 요청 → [응답 대기] → refreshGalleryData()
                                          → /api/items + /api/tags + /api/workflows 3개 전부 재fetch
                                          → 전체 갤러리 리렌더
```

문제 지점 셋:
1. **응답 대기 후에야 화면 반영** — optimistic update 없음. 서버 왕복 시간만큼 UI가 멈춘 듯 보임.
2. **변경 하나에 3개 API 전체 재조회** — 태그 하나 바꿔도 items·workflows까지 다시 가져옴.
3. **버튼 `disabled={saving}`** — 응답 올 때까지 버튼이 잠겨 "먹통" 인상 (모달 즐겨찾기 `ItemModal.tsx:211`).

이건 개별 버그가 아니라 **데이터 흐름 설계**입니다. 그래서 한 곳만 고치면 나머지가 그대로 남습니다.

## 3. 개선 전략 — 구조 하나를 고치면 전부 해결

개별 핸들러를 9개 따로 고치는 게 아니라, **공유 상태 갱신 계층을 바꿉니다.**

### 핵심: 전체 재fetch → 로컬 상태 부분 갱신

`refreshGalleryData()`(전체 재조회)를 기본 경로에서 걷어내고, 변경된 데이터만 로컬 `galleryData` state에 직접 반영하는 헬퍼로 대체합니다.

```
// 지금: 무조건 전체 재조회
await mutate(); await refreshGalleryData()

// 개선: 로컬 상태 즉시 패치 (optimistic) → 서버는 백그라운드 → 실패 시 롤백
setGalleryData(patchLocally(prev))   // 즉각 반영
mutate().catch(() => setGalleryData(prev))  // 실패 시 되돌림
```

### 우선순위 (Wave 2 안에서 단계)

- **2-a (필수)**: 즐겨찾기(1·2) optimistic + 버튼 disabled 제거 → 체감 최대
- **2-b (필수)**: 카드 태그 추가/삭제(3) optimistic
- **2-c (권장)**: 생성/수정/삭제(4·5)는 낙관 반영이 위험하므로(서버 파생값 존재), **전체 재fetch 대신 해당 항목만 재조회**로 완화
- **2-d (선택)**: 태그 관리 모달(8) 행별 낙관 반영

### 왜 4·5는 다르게 다루나 (트레이드오프)

생성·수정 시 서버가 **title 자동생성·자동 태그 부착·updatedAt** 등 파생값을 만듭니다(§Wave3의 E 항목과 연결). 클라이언트가 이를 미리 알 수 없어 순수 optimistic이 위험합니다. → 이들은 "즉시 반영"보다 "**단일 항목만 재조회**"로 왕복·리렌더 범위를 줄이는 게 안전합니다.

## 4. 횡단 개선 — 이번에 함께 정리할 중복/구조 문제

사용자 요청 "구조 개선 시 다른 코드에도 적용할 부분"에 해당하는, 진단 중 발견한 중복들:

### 4-1. `errorMessage` + `submitJson` 3중 복붙 (확실한 중복)
`item-mutations.ts`, `tag-mutations.ts`, `workflow-mutations.ts` 세 파일에 **완전히 동일한** `errorMessage`(20줄)와 `submitJson`이 각각 복사돼 있습니다. → 공용 `api-client.ts`로 추출. Wave 2에서 optimistic 배선을 만지는 김에 함께 정리하면 자연스러움.

### 4-2. 상태 갱신 로직이 App.tsx에 산재
`toggleFavorite`·`changeTags`가 각자 "재조회 후 modalState 동기화"를 중복 구현(App.tsx:87-114). optimistic 도입 시 **단일 `updateItemInState(id, patch)` 헬퍼**로 통합하면 modalState 동기화도 한 곳에서 처리.

### 4-3. `disabled={saving}` 패턴 점검
모달 즐겨찾기만 `disabled`가 걸려 있고 카드는 없음 — 일관성 없음. optimistic 전환 후엔 **양쪽 다 disabled 불필요**(즉시 반영되므로). 저장/삭제처럼 되돌리면 곤란한 동작에만 남김.

## 5. Wave 2 재정의 (확장본)

기존 Wave 2 = "즐겨찾기만" → **"반응성 구조 개선"으로 확장**:

| 하위 | 내용 | 성격 |
|---|---|---|
| 2-a | 즐겨찾기 optimistic + disabled 제거 | 체감 최대 |
| 2-b | 카드 태그 add/remove optimistic | 체감 높음 |
| 2-c | 생성/수정/삭제 → 단일 항목 재조회로 완화 | 안전한 완화 |
| 2-d | 태그 관리 모달 행별 낙관(선택) | 저빈도 |
| 2-e | `api-client` 추출 (errorMessage/submitJson 중복 제거) | 횡단 정리 |
| 2-f | `updateItemInState` 상태 헬퍼 통합 | 횡단 정리 |

### 검증 방법 (완료 기준)

- 즐겨찾기 클릭 시 **네트워크 응답 전에** 별 아이콘이 즉시 채워진다 (DevTools Network throttling으로 확인).
- 태그 추가 시 전체 갤러리 리렌더가 아니라 해당 카드만 갱신된다.
- 서버 실패(강제 500) 시 낙관 반영이 롤백되고 에러가 표시된다.
- 기존 worker 테스트(`pnpm test:worker`)가 통과한다 (API 계약 불변).
- 3개 mutation 파일의 중복 코드가 사라지고 타입체크(`pnpm typecheck`)·린트(`pnpm lint`) 통과.

## 6. 착수 전 결정 필요 (인터뷰 항목)

- **2-c 범위**: 생성/수정/삭제를 "단일 항목 재조회"로 완화 vs 지금처럼 전체 재fetch 유지(단순함 우선)
- **2-d 진행 여부**: 저빈도인 태그 관리 모달까지 낙관 반영할지
- **롤백 UX**: 낙관 반영 실패 시 조용히 되돌릴지, 토스트로 "실패, 되돌림" 알릴지
