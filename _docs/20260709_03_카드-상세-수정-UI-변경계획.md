# 카드 상세 수정 UI 변경계획

## 결론

현재 HTML 미리보기는 논의안을 반영한 제안 화면이고, 실제 제품 코드는 아직 대부분 이전 구조다. 구현 계획은 미리보기를 그대로 복사하는 방식이 아니라, 기존 React 컴포넌트 구조에 맞춰 `GalleryCard`, `ItemModal`, `ItemModalForm`, `ItemModalDetail`, `ItemModalActions`, 카드/모달 CSS를 정리하는 방향이 맞다.

## 선행조건

- 자동/수동 태그(`manual`/`auto` source) 작업이 완료된 상태에서 착수한다. 이 작업은 완료되었다.
- UI 변경은 태그 source semantics가 확정된 기준 위에서 진행하며, 태그 칩 편집은 manual source만 편집하고 auto source는 읽기 전용으로 표시한다.
- 착수 전 태그 변경 완료 상태를 기준점으로 고정한다(커밋 또는 작업트리 기준 확정).

## 확정된 UX 방향

- `Item detail`, `항목 수정`, 읽기 전용 `유형` 필드는 과하다.
- 유형은 수정 가능한 필드가 아니라 상단의 작은 배지로 처리한다.
- 제목은 선택값이므로 본문보다 먼저 나오면 안 된다.
- 자동 제목/본문 첫 줄 제목 중복은 카드에서 숨기지 않고 기존과 동일하게 둔다. 카드 규칙을 단순하게 유지하고, "왜 어떤 카드만 제목/preview가 다른가"라는 혼란을 피하기 위해 별도 숨김/생략/정규화 로직은 추가하지 않는다.
- 카드는 블로그 게시글이 아니므로 날짜를 강조하지 않는다.
- 날짜가 필요하면 상세/수정의 하단 왼쪽에 최종 수정일만 `YYYY-MM-DD`로 둔다.
- 상세와 수정 모달은 구조가 크게 달라지면 안 된다.
- 복사/즐겨찾기/X는 우측 상단 아이콘 액션으로 통일하며, 상세뿐 아니라 수정 모달에도 복사/즐겨찾기를 포함한다.
- 수정/삭제/저장/취소 같은 상태 전환 액션은 하단에 둔다.
- 추가 모달은 상단 3개 유형 선택 UI를 제거하고, 섹션별 고정 유형 배지로 유형을 표시한다.
- 태그는 칩 단위로 바로 삭제/추가 가능해야 하며 상세/수정 모두 동일해야 한다. 단, auto 태그(예: `github`)는 삭제 버튼 없는 읽기 전용 칩으로 표시한다.
- 워크플로우 카드는 실행 기능이 없으면 플레이 버튼을 쓰면 안 된다. 즐겨찾기도 도입하지 않고, 상단 액션은 `스텝 목록 복사`만 둔다.
- 레포 카드는 상단에 `GitHub 열기` 액션을 유지한다.

## 변경계획

1. `GalleryCard` 구조 정리
   - 프롬프트/레포/워크플로우 카드는 `상단 배지+액션 -> 내용 영역 -> 제목 -> 태그` 순서로 맞춘다.
   - 이미지 카드는 기존 이미지 자체가 내용 영역이므로 별도 텍스트 영역을 추가하지 않는다.

2. 워크플로우 카드 액션 정리
   - 워크플로우는 실행 버튼도 즐겨찾기도 두지 않고 `스텝 목록 복사` 액션만 유지한다.
   - 즐겨찾기는 이번 범위에서 제외한다. 현재 `WorkflowItem`에는 `favorite` 필드와 전용 토글 API가 없어, 도입하려면 저장 구조·API·즐겨찾기 탭 포함 여부까지 별도 설계가 필요하기 때문이다. 후속 항목은 백로그 `BL-0709-01`로 분리한다.

3. 카드 날짜 제거 또는 축소
   - 카드 footer의 날짜는 제거한다.
   - 레포의 `GitHub 열기`는 카드 상단 우측 액션 그룹에 둔다. 복사/즐겨찾기와 같은 즉시 액션이고, 날짜 제거 후 하단은 태그/보조 정보만 남기기 때문이다.

4. `ItemModal` 헤더 재구성
   - `Item detail/New item/항목 수정` 텍스트를 제거한다.
   - 상단은 `유형 배지 + 우측 아이콘 액션(복사, 즐겨찾기, 닫기)`로 통일한다.

5. 상세/수정 공통 레이아웃 정리
   - 상세는 `본문 -> 제목 -> 태그 -> 메모/링크 -> 하단 날짜+수정/삭제`로 맞춘다.
   - 수정은 `본문 -> 제목(선택) -> 태그 -> 메모 -> 하단 날짜+삭제/취소/저장`로 맞춘다.

6. 수정 모달 액션 보강
   - 현재 `EditActions`는 취소/저장뿐이므로 삭제 버튼과 상단 복사/즐겨찾기 액션을 유지할 수 있게 props 구조를 바꾼다.
   - 상단 복사/즐겨찾기는 상세 모달과 동일하게 수정 모달에도 포함한다(상세/수정 구조를 크게 다르게 만들지 않는다).
   - 수정 모달의 복사 대상은 현재 draft 본문 기준으로 한다. 수정 중에는 사용자가 보고 있는 편집 내용을 복사하는 게 자연스럽기 때문이다.
   - 즐겨찾기는 저장 여부와 무관한 item 상태 토글이므로 수정 모드에서도 유지한다.
   - 하단 버튼 순서는 기존과 동일하게 둔다.

7. 태그 칩 편집 반영
   - 현재 수정 폼은 쉼표 입력 방식이다.
   - 논의안대로 하려면 `tagsText` 단일 input 대신 칩 리스트 + `+ Tag` 입력/삭제 UI로 바꿔야 한다.
   - auto 태그는 삭제 버튼 없는 읽기 전용 칩으로 렌더링해 사용자 태그와 구분한다. manual 태그만 추가/삭제 가능하다.
   - UI 내부 상태는 칩 배열이어도 submit payload는 기존 `tags: string[]` 형태를 유지한다. 백엔드/API payload 구조는 바꾸지 않는다.

8. 추가(입력) 모달 유형 선택 정리
   - 추가 모달 상단의 3개 유형 선택 UI는 제거한다.
   - 섹션별 추가는 유형 선택 없이 해당 섹션 고정 유형 배지로 표시한다.
   - 유형 선택이 필요한 공통 "전체 추가" 진입이 남아야 한다면, 그 별도 진입에서만 유형 선택을 노출한다.

9. CSS 검수
   - `src/client/styles/cards.css`, `src/client/styles/modal.css`에서 카드 내용 영역, 제목 clamp, 모달 footer 한 줄 배치, 모바일 폭에서 버튼 줄바꿈을 조정한다.
   - `DESIGN.md`의 카드/모달 원칙은 유지하되, 논의된 카드형 상세/수정 규칙은 `DESIGN.md`에도 짧게 반영하는 게 맞다.

## 완료 기준

- 카드 footer에 날짜가 보이지 않는다.
- 프롬프트/레포/워크플로우 카드는 본문(내용 영역)이 제목보다 위에 온다.
- 상세/수정 모달 모두 본문이 제목보다 먼저 나온다.
- 상세 모달과 수정 모달 모두 상단에 복사/즐겨찾기 액션이 있다.
- 워크플로우 카드 액션은 `스텝 목록 복사`만 있고 즐겨찾기가 없다.
- 레포 카드는 상단 우측에 `GitHub 열기` 액션이 있다.
- auto 태그 칩에는 삭제 버튼이 없고, manual 태그 칩만 추가/삭제된다.
- 추가 모달에 상단 3개 유형 선택 UI가 없다.
- 375/768/1280px에서 가로 넘침이 없다.

## 논의 반영 이력 (2026-07-09)

이 계획은 HTML 미리보기 검토 세션에서 아래와 같이 조정되었다.

- 자동 제목/본문 첫 줄 중복 숨김 로직은 도입하지 않고 기존 방식을 유지하기로 확정했다(카드 규칙 단순성 우선).
- 워크플로우 카드에는 즐겨찾기를 두지 않고 `스텝 목록 복사`만 유지한다. 즐겨찾기는 데이터/API 설계가 선행되어야 하므로 백로그 `BL-0709-01`로 분리했다.
- auto 태그는 삭제 버튼 없는 읽기 전용 칩으로 표시하고, manual 태그만 추가/삭제한다.
- 수정 모달에도 상단 복사/즐겨찾기 액션을 포함한다(제외 아님). 복사는 draft 본문 기준, 즐겨찾기는 저장 여부와 무관한 item 상태 토글이다. 세션 중 "제외" 정리는 오류였고 "포함"으로 정정 확정했다.
- 레포 카드의 `GitHub 열기`는 카드 하단이 아니라 상단 우측 액션으로 확정했다(미리보기 기준 정정).
- 추가 모달은 상단 3개 유형 선택 UI를 제거하고 섹션별 고정 배지로 처리한다.
- 태그 submit payload는 기존 `tags: string[]`를 유지하고 백엔드는 바꾸지 않는다.
- 선행조건: 자동/수동 태그 작업 완료 후 착수(완료됨).
