이 계획은 클라이언트 UI 문제 개선 및 반응성 향상이라는 **핵심 목적을 달성할 가능성이 높으나, 현재 상태로는 `done-check`를 통한 자동 검증 및 무인 실행이 불가능**합니다. 실행 전 아래 지적된 문제들을 수정해야 합니다.

### 1. 자동화 검증(done-check) 한계 및 Headless 실행 부재 (가장 크리티컬)
- **문제점**: 계획의 검증 방법이 100% 수동(육안 확인, 스크린샷, DevTools 네트워크 탭 확인)에 의존하고 있습니다.
- **Headless 부재**: `pnpm qa:browser`에 대해 "또는 수동 브라우저 검증"이라는 회피성 문구가 포함되어 있으며, Playwright headless 실행이 명시되어 있지 않습니다. 브라우저 창이 뜨면 사용자 포커스를 강탈하며, `done-check`가 읽을 수 있는 시스템적 증거(종료 코드, JSON 로그)가 남지 않습니다.
- **수정 요구**: 육안 확인과 수동 검증을 모두 Playwright headless 테스트로 교체하고, "테스트 통과 및 리포트(증거 파일) 생성"을 완료 기준으로 명시해야 합니다.

### 2. 실행 불가 Placeholder 및 모호성
실행자가 판단해야 하는 미확정 요소가 다수 존재하여, 이대로는 실행 중단(블로킹)이 발생합니다.
- **1-2**: `실패 피드백 대안 확인` → 실행 전에 실패 피드백 대안(예: 에러 토스트 띄우기)을 확정해야 합니다.
- **3-2**: `A 구현 시 실제 브라우저 확인 후 확정` → Masonry 구현 방식을 실행 시점에 결정하라고 미루고 있습니다. CSS `column-count`로 갈지, JS 기반 Grid로 갈지 문서 단계에서 확정해야 합니다.
- **3-3**: `줄 수(2/3)는 masonry 확정 후 결정` → 2줄로 할지 3줄로 할지 명확한 값을 지정해야 합니다.
- **3-5**: `타이틀 적절히 숨김` → "적절히"는 검증할 수 없는 단어입니다. `특정 브레이크포인트 이하에서 display: none 적용`으로 명확히 해야 합니다.

### 3. 검증 산출물 처리 누락
- 스크린샷을 찍으라는 내용(1-1, 1-4, 3-2 등)이 있으나, 스크린샷이 **어느 디렉토리**에 **어떤 파일명**으로 저장되어야 `done-check`가 이를 증거로 사용할 수 있는지 명시되지 않았습니다. 산출물의 라이프사이클이 닫혀 있지 않습니다.

### 4. Wave 2 Optimistic Update 트레이드오프 검토
- 전체 API fetch 대신 단건 재조회(2-c)로 서버 파생값(title 등) 문제를 해결한 것은 훌륭한 트레이드오프입니다.
- **결함(Race Condition)**: 2-b에서 "자동 태그는 서버 응답 후 병합"한다고 명시했으나, 사용자가 연속으로 태그를 추가/삭제할 때 늦게 도착한 서버 응답이 로컬 상태를 덮어씌워 UI가 튀는 현상(순서 꼬임)에 대한 대처 방안이 없습니다.

### 5. Wave 3-2 Masonry 읽기 순서/접근성 리스크
- `column-count` 방식을 사용할 경우 DOM 순서는 상하, 시각적 읽기 순서는 좌우가 되어 탭(Tab) 이동 순서가 100% 엇갈리게 됩니다.
- 검증 기준에 `키보드 Tab 순서가 시각 순서와 크게 어긋나지 않는지 확인`이라고 적혀있으나, `column-count`를 쓰면 **무조건 어긋나므로 이 검증은 실패할 수밖에 없습니다.** 이를 접근성 결함으로 감수할지, 탭 순서를 JS로 보정할지 명확한 결정이 필요합니다.

### 6. 과최적화(Over-optimization) 징후
- **2-e (api-client 추출)**: 3개 파일의 에러 메시지 20줄을 공용 모듈로 분리하는 것은 유용하지만, UI 반응성 개선이라는 현재 Wave의 핵심 목표와 무관한 **순수 리팩토링(과최적화)**입니다.
- 목적 중심 실행을 방해할 수 있으므로, 변경 범위를 넓히는 2-e 작업은 후순위(Wave 4 이후)로 미루거나 삭제하는 것을 권장합니다.
